# 更新日志

本文件记录用户可见的变更。格式参考 [Keep a Changelog](https://keepachangelog.com/zh-CN/1.1.0/)，
版本号遵循 [语义化版本](https://semver.org/lang/zh-CN/)。

## [1.0.10] — 2026-07-15

顶栏自动提示新版本；并为「朋友在 Mac 上从源码跑」做了跨平台适配（Windows 行为不变）。

### 新增
- **右上角自动更新提示**：有新版本时，顶栏右上角（⚙ 设置 左边）会出现「🎁 有新版 vX →」，
  点一下直接进设置的更新区一键升级。启动时会静默检查一次（设置 → 应用更新 里有「启动时自动检查
  新版本」开关，**默认开**，不想让它每次启动连 GitHub 就关掉）。右键徽标可忽略当前版本，
  发更新的版本时会自动重新提醒。
- **macOS 从源码运行支持**（面向个人自用，非公开发布的打包）：数据落点改到
  `~/Library/Application Support/PaperPiggy`、依赖按平台分流（macOS 装 pyobjc、跳过 Windows 专属的
  pythonnet）、Zotero 目录探测支持 macOS profile、打开文件夹/取消建库等改为跨平台。
  详见 [docs/MAC-从源码运行.md](docs/MAC-从源码运行.md)。**以上改动在 Windows 上一律无变化。**

## [1.0.9] — 2026-07-15

为「重装一次库、深索重来一遍」铺路：只导 PDF 更严格、深索不再假装卡在 99%、并新增一键清空重建。

### 修复
- **深索到 99% 就「卡住不动」（其实没卡）**：所有有正文的文献都嵌完之后，还有最后一步——
  重建 bm25 检索索引（几十万个文本块，分词+建索引要好几分钟）。这期间进度条冻在 99%、
  深索计数也不动，看着像死机，其实在收尾。现在这一步会明确显示「**深索已完成，正在重建检索索引（收尾中）**」，
  不再误导你以为卡死。
- **「只导入有 PDF 的文献」以前会放行法源类**：法规/案例/标准/报告即使没有 PDF 也照样进库，
  和开关字面意思不符、也让一堆无正文的纯题录混进来。现在**严格只导有 PDF 的**——
  重要法条你自己配上 PDF 再入库即可。

### 新增
- **一键「清空并从头重建索引」**（设置 → 🧹 清空并从头重建索引）：想把索引推倒重来时用。
  它**只清索引**（全文提取、切块、向量、bm25、深索标记、整库状态），
  **明确保留**综述库、收藏夹分类、期刊分级、检索摘要（花过 API 钱的）、页码映射、设置、
  `0_Agent` 工作区。清空后点「⟳ 更新知识库」重建题录+语义（会**自动去重、按“只导 PDF”重新导入**），
  再「深索全部」重新深索。旧的提取/切块会移到一个备份目录、可手动捞回。
  ⚠️ 向量索引会被丢弃、需重新嵌入，操作前建议先在「💾 备份与恢复」打一份含索引的备份。

## [1.0.8] — 2026-07-15

一批体验与正确性修复：浏览标签刷新、深索面板、导入去重、升级不再闪黑窗、充值提示。

### 修复
- **深索过程中浏览页仍显示旧的「扫描件」标签**：文献早已重新提取入库、后端也已实时更正，
  但浏览列表加载一次后整段深索期间不刷新。现在深索/自愈有进展时，浏览徽标会**自动跟上**
  （不动你的勾选、滚动、分页）。
- **深索详情面板「整库深索中 / 已暂停 / 暂无正在深索」三态混显**：暂停按钮、「正在深索」列表
  是**队列**的概念，整库深索时不适用却还显示。现在整库深索时只留干净的进度 + 「停止整库深索」。
- **应用内升级时会闪一下终端黑窗**：更新器的一个子进程漏了「无窗」标志。已修，并把全项目
  所有子进程统一无窗（以后新功能也默认无窗）。

### 新增
- **导入去重**：Zotero 里同一篇被重复添加（不同条目）会各占一行，现在入库时自动合并。
  ⚠️ **绝不删已深索/已入库的副本**——去重只清没有任何深索产物的纯题录重复，你花 API 钱嵌好的
  一份永远保留。（对已全部处理过的老库不会动手，主要防将来新导入的重复。）
- **充值提示**：设置 → 检索引擎处提醒——SiliconFlow 账户余额为 0 时连免费模型也会报 403，
  建议先充 ¥1，之后免费模型即可一直正常用。

## [1.0.7] — 2026-07-15

自愈 1.0.6 之前那次崩溃留下的「僵尸扫描件标记」——1.0.6 修好了 PDF 提取，但之前被误判的
文献仍卡着进不了深索、浏览还显示「扫描件」。本版让它们自动补回。

### 修复
- **修好提取后，文献仍显示「🚫 扫描件·需 OCR」、深索进不去**：崩溃期大批 PDF 假失败时被
  打了 `no_text`（扫描件）标记，这个标记是**粘性**的、从不复查——即便后来 PDF 已重新提取、
  切块成功，深索仍永远跳过它们、浏览仍显示扫描件。现在深索前会**对账自愈**：凡标了扫描件、
  但其实已切出真实正文块的，一律清除标记、补入深索。真正无正文的扫描件仍正确保留标记。
  （升级后点一次「深索全部」即可把这些文献补回；真正的扫描件不受影响。）

## [1.0.6] — 2026-07-15

修复深索时应用崩溃 + 大量文献被误判为「扫描件」——根因是 PDF 提取库的线程不安全。

### 修复
- **深索时整个应用窗口崩退**：PDF 提取用 4 线程并发调 PDFium，而 **PDFium 不是线程安全的**，
  多线程同时访问会损坏其内部堆（Windows 事件日志确诊：故障模块 `pdfium.dll`，
  异常码 `0xc0000374` 堆损坏 / `0x80000003`）。改为**所有 PDF 提取全局串行**，绝不并发。
  实测：同一批 500 个 PDF，并发时约 70% 失败/偶发崩溃，串行后 100% 成功、不崩。
- **大量有文字的文献被误标「🚫 扫描件·需 OCR」**：正是上面并发竞态导致的随机
  「Failed to load」（同一 PDF 单独打开明明好好的），被误判成扫描件、深索篇数虚低。
  串行修复后，重跑「深索全部」会自动重试这些篇并成功入库。
- PDF 提取加瞬时失败重试（文件被 OneDrive / 杀软 / Zotero 短暂占用时多试两次）。

> 升级后请重新点一次「深索全部」：之前因崩溃/误判没入库的文献会自动补上（不会重复劳动，已成功的跳过）。

## [1.0.5] — 2026-07-15

修复 API 模式下批量深索/建索引的崩溃、假完成与卡死。

### 修复
- **API 模式深索全库会崩**：嵌入调用没有兜底，一次 API 失败（余额不足→403、限流、网络超时）
  就会带 traceback 掀翻整个后台任务。现在：致命错（密钥/余额/模型名）立即优雅中止并给人话提示；
  限流/网络类跳过该篇、连续多次才判停；**任一篇没嵌成功都不会谎报完成**，已嵌的部分照常保存、可续跑。
- **崩了却提示「✓ 深索完成」**：失败后紧接着的后台任务会把全局退出码覆写掉，前端据此误报成功。
  现在后端失败时不再立即接排新任务、前端只认「退出码 0」的阳性证据才报完成，否则一律「⚠ 深索中断」。
- **进度条卡死在「正在提升检索质量 X/Y」**：语义嵌入中途失败后，进度条会永远停在那儿谎报进行中。
  现在区分「真在跑」（动进度条）与「上次未完成」（静态提示「点『手动更新知识库』继续」）。

## [1.0.4] — 2026-07-15

修复应用内更新的一个致命 bug，并增强国内下载可靠性。

### 修复
- **应用内更新点「下载并升级」报 `name 'threading' is not defined`**：启动器缺 `import threading`，
  升级的最后一步（拉起更新进程 + 关窗）直接崩，导致 1.0.2 / 1.0.3 的应用内更新**根本用不了**。
  已修。⚠️ 因为这个 bug 就在旧版里，**1.0.2 / 1.0.3 用户须用安装器手动装一次 1.0.4**；此后即可应用内一键升级。

### 新增 / 变更
- **下载重试**：检查更新、下载增量包都改为失败自动重试几次——国内连 GitHub 偶发超时，多试往往就通。
- **国内镜像兜底**（⚙ 设置 → ⬆️ 应用更新 → 国内镜像）：填一个镜像目录前缀（Gitee Release / 对象存储），
  应用内更新会先试 GitHub、连不上就走镜像。只需镜像那个约 1MB 的增量包。设置方法见 README。

## [1.0.3] — 2026-07-15

调整三项出厂默认（只影响**新安装**；已装用户的现有设置保持不变，可在设置里随时改）。

### 变更
- **默认「只导入有 PDF 的文献」**：Zotero 里没有 PDF 的纯题录不进库，库更干净、卡片更完整。
  ⚠️ 代价：没有 PDF 的法条/法规/网页法源会被挡在库外——需要收全法源的，在
  「⚙ 设置 → 只导入有 PDF」里关掉即可。
- **期刊分级学科默认「法学（开发者增强）」**：含台湾刊 / 顶尖外文法评 / 外文权威社科刊，
  外刊不打折。标准法学分级仍可在设置里切回。
- **深索摘要默认「交给 Agent 生成」**：省 API 额度、契合以 Agent 为主的用法。
  注意：应用内自己点「深索」时不会自动补摘要，需让 Agent 跑深索时顺带生成、或在
  浏览页/库总览手动补。想让应用自动生成，在设置里切到「服务端生成」并填 Key。

## [1.0.1] — 2026-07-15

修复首个版本的两个启动问题，并重定发布形态。[下载](https://github.com/DrinkTea905/paper-piggy/releases/tag/v1.0.1)

### 修复
- **桌面快捷方式点了没反应**：启动器 `PaperPiggy.vbs` 用 `SW_HIDE` 启动子进程，
  被 pywebview 继承成「隐藏窗口」——应用其实每次都启动成功，只是窗口看不见，
  用户每点一次就多一个后台幽灵进程。快捷方式改为直接指向 `pythonw.exe`，绕开 VBS。
- **重复启动**：应用没有单实例保护，双击 N 次会起 N 个进程抢同一端口。
  现在再次启动会把已有窗口拉到前台、自己退出。
- **Agent 模板旁本 `.new.md` 被静默覆盖**：提示用户「对照合并」，用户在旁本里写的
  合并笔记会被下次启动的出厂原文盖掉。现在旁本被改过就另存为 `.new.2.md`，绝不覆盖。

### 变更
- **只发 Inno 安装器，不再出便携 zip**。数据与程序同目录后，便携版「删旧文件夹、
  解压新版」的常规升级姿势会把索引、综述、写好的论文一次性删光；安装器升级不会
  （只覆盖程序本体）。
- **改为用户级安装**（`%LOCALAPPDATA%\Programs\PaperPiggy`，可在向导里改到任意盘），
  安装不再需要管理员权限、不弹 UAC。
- **数据与程序放同一个文件夹**：索引、模型、综述、Agent 交付物全在安装目录内，
  整个文件夹拷走即可换机。安装目录不可写时自动回退到 `%LOCALAPPDATA%\PaperPiggy`。
- 用户数据目录由 `%LOCALAPPDATA%\LocalKB` 改名为 `PaperPiggy`。
- **更新方式说明**：目前没有应用内自动更新；升级 = 到 GitHub 下新版安装器覆盖安装，
  只覆盖程序本体（`app\`、`python\`），索引、综述、写好的论文等数据不受影响。
  （注意与设置里的「自动更新知识库」区分——那是知识库增量索引，不是版本升级。）

### 新增
- **备份与恢复**（设置 → 💾 备份与恢复）：把综述、Agent 写的论文与记忆、收藏夹、
  期刊分级等「丢了就再也没有的东西」打成一个 zip。备份位置可指到 OneDrive／坚果云
  的文件夹实现云备份（同步静态 zip 是安全的，而云盘直接同步向量索引会把它搞坏）。
  可选连向量索引一起打包（换机免重建）；可设自动备份。API 密钥默认不进备份包。

## [1.0.0] — 2026-07-14

首个公开版本。**已撤下**——存在启动窗口不显示的 bug（见 1.0.1 修复），release 与其安装包已删除，请用 1.0.1。

### 新增
- 项目指引本地化：`CLAUDE.md`（AI agent 总纲）、`docs/ARCHITECTURE.md`、`docs/MAINTENANCE.md`
- 开源三件套：`LICENSE`（Apache-2.0）、`THIRD-PARTY-NOTICES.md`、`CHANGELOG.md`
- git 版本控制（此前靠 15 个手工备份目录，已回放为 git 历史）

### 变更
- **PDF 提取引擎改用 `pypdfium2`**（原 `pymupdf4llm`）。原方案会间接引入 `pymupdf_layout`
  （Polyform Noncommercial：禁止商业使用、非 OSI 开源）与 AGPL 的 PyMuPDF，
  与本项目的 Apache-2.0 冲突，**导致此前根本无法合法开源发布**。
  实测替换后：中文提取字符数逐篇完全相同（无内容损失），文本质量更好
  （原方案会把中文标点错序），单篇提取快 20~100 倍。
- `requirements.txt` 的 `transformers` / `tokenizers` 版本上限修正为与实装一致
  （此前写 `<5` / `<0.22`，而实装是 5.13.0 / 0.22.2，照着重装会得到一个从未验证过的组合）。

### 修复
- **Word 导出静默降级**：`python-docx` 声明在 requirements 却从未装进分发包，
  导致综述导出恒降级为 `.md` 且不报错。
- **干净机上本地模式必崩**：分发包的 Python 运行时缺 `msvcp140.dll` / `msvcp140_1.dll`
  （`onnxruntime` 的硬依赖），在未装 VC++ 2015-2022 的机器上 import 即 `WinError 1114`。
- MCP 工具表漂移：文档写「28 个工具」，代码里实际 32 个（`gen_mcp_doc.py --check`
  本可检出，但从未被调用）。
- `.claude/launch.json` 与 MCP 配置指向已不存在的目录。

### 移除
- 分发包目录 `LocalKB/`（2.1G 测试数据与构建产物）。构建改为从源码 + `build/` 资产生成。

### 打包
- Inno Setup 安装器（177 MB）+ 便携 zip（282 MB）+ updater 用的增量包
- 模型（约 900 MB）走 GitHub Release 按需下载，支持断点续传与 sha256 校验；API 模式无需下载
- 缺 WebView2 时安装器静默安装 Evergreen Runtime
