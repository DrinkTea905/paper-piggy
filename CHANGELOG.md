# 更新日志

本文件记录用户可见的变更。格式参考 [Keep a Changelog](https://keepachangelog.com/zh-CN/1.1.0/)，
版本号遵循 [语义化版本](https://semver.org/lang/zh-CN/)。

> 维护约定：**每次发版必须更新本文件**（见 [docs/MAINTENANCE.md](docs/MAINTENANCE.md) 的 checklist）。
> 版本号的唯一事实源是 `config.APP_VERSION`。

---

## [1.0.21] — 2026-07-17

应用更新后，用户自己改过的工作流和综述库规约不再只是静默留下一份 `.new.md`：现在会在
Agent 页明确提示，并提供差异查看与安全合并入口。

> **升级方式**：本版没有新增运行组件，v1.0.20 用户可直接使用应用内更新。仍在较早版本，
> 或应用提示“需完整安装器”的用户，请运行 v1.0.21 完整安装器覆盖原位置；不用卸载，文献索引、
> 模型、综述库和 `0_Agent资料库` 都会保留。

### 改进
- **用户改过的工作流不再悄悄错过新版** —— 未修改的出厂工作流仍会自动升级；用户修改过的文件
  保持原样，新版另存旁本，并在 Agent 页显示“待合并”。可查看逐行差异、复制一份合并任务给
  Agent，或在自动留下 `user-backup` 后直接采用新版。
- **Agent 会主动知道哪些规则待合并** —— MCP 接入指令会列出待处理的工作流与 Wiki 规约，要求
  Agent 保留用户个性化规则、合并新版新增要求，遇到冲突先询问，不能擅自覆盖。
- **增加统一升级健康检查** —— 同时检查 Agent 模板、综述库规约、索引生成规则、Python 运行环境
  和本地模型版本；应用代码更新了、但配套内容仍停在旧版时，会明确说明应采取的动作。

### 安全性
- **增量更新增加运行环境指纹** —— 未来新版若需要不同的 Python 运行组件，应用内更新会在替换
  程序前安全停止，并引导使用完整安装器，避免“代码更新了、运行环境没更新”的半升级状态。
- **发布护栏覆盖全部出厂模板** —— 12 份 Agent 模板的当前版本 hash、Wiki schema 与正文版本
  都会在打包前自动核对，漏登记将直接中止构建。

## [1.0.20] — 2026-07-17

浏览页现在能直接看清每类文献有多少篇；设置页重新分组后，也更容易找到检索、建库、备份和更新功能。

> **升级方式**：本版没有新增运行组件。已通过 v1.0.17 或更新版本的完整安装器补齐 OCR 组件的用户，
> 可以直接使用应用内更新；仍在更早版本，或应用提示“需完整安装器”的用户，请运行 v1.0.20
> 完整安装器覆盖原位置，不用卸载，用户数据会保留。

### 改进
- **筛选项直接显示篇数** —— “已深索 / 未深索、OCR / 原生文字层、摘要有效 / 异常 / 缺失”都会
  显示当前左侧范围内的文献数量；切换分类、收藏夹或时间范围时，数量同步变化。
- **摘要状态更容易辨认** —— 使用有颜色且含义稳定的“🧬 摘要有效、🟠 摘要异常、⚪ 摘要缺失”，
  便于快速定位需要处理的文献。
- **设置页按任务分成五类** —— 检索、建库、外观、数据与维护、应用分别放置，原有设置和数据不变；
  两套应用内教程及相关跳转路径已同步更新。
- **Agent 交付物卡片更整齐** —— 日期与文件统计固定分两行显示，不再因内容多少造成卡片下方大片空白。

### 修复
- 老版本已经深索、但没有记录“正文来源”的文献，会按历史事实归入“原生文字层”，不再造成
  原生文字层筛选只显示极少数新文献。

## [1.0.19] — 2026-07-17

深索结果与检索摘要现在更容易核对：浏览页可直接筛出 OCR 文献和摘要异常；Agent 写论文、
做综述时则会先覆盖更多文献，再对选定材料深入取证。

> **升级方式**：本版没有新增运行组件。已通过 v1.0.17 或更新版本的完整安装器补齐 OCR 组件的用户，
> 可以直接使用应用内更新；仍在更早版本，或应用提示“需完整安装器”的用户，请运行 v1.0.19
> 完整安装器覆盖原位置，不用卸载，用户数据会保留。

### 改进
- **浏览页可直接筛 OCR 与检索摘要状态** —— 原有筛选新增“本地 OCR / 原生文字层”和
  “有有效摘要 / 摘要异常 / 摘要缺失”，可直接定位扫描件和需要修复或补生成摘要的文献。
- **检索摘要增加严格质量闸门** —— 乱码、连续重复、失控长文和“没有正文”的过期占位摘要不再计为完成，
  也不再用于新的嵌入；界面和 MCP 会分别显示有效、异常、缺失数量及异常原因。只有用户主动点
  “修复 / 补生成摘要”时才会重写并重嵌入。
- **Agent 检索改为先广后深** —— 发现型检索同一篇最多返回 2 段，候选不足时不再用同一篇的重复段凑数；
  `verify_claim(keys=[...])` 等定向核验仍可在已经选定的文献内多段取证。内置“写论文与综述”和
  “跨学科发散与补文献”工作流已同步这一用法。

### 修复
- 深索队列已清空时，MCP 不再误报“运行中”；PDF 缺失、损坏、OCR 等提取异常会单列显示。
- `read_source` 只读指定页时不再把该页误报成全文末页；论断核验不再返回空引文或重复证据。

## [1.0.18] — 2026-07-17

升级安装不再把 Codex、Claude 等客户端持有的 PaperPiggy MCP 笼统显示成一串“Python”，
也不会在后台连接仍存活时尝试强行覆盖程序文件。

> **升级方式**：已通过 v1.0.17 完整安装器补齐运行组件的用户，无需为本修复重新覆盖安装，
> 可以继续使用应用内更新。仍在 v1.0.16 或更早版本，或者虽已增量更新到 v1.0.17 / v1.0.18
> 但仍提示“需完整安装器”的用户，请直接运行 v1.0.18 完整安装器覆盖原位置；不用卸载，
> 用户数据会保留。本版的 MCP 占用预检修复只存在于完整安装器，增量包不会改变安装器行为。

### 修复
- **升级时明确指出是谁占用了程序文件** —— 完整安装器会在写文件前识别 PaperPiggy 主程序，
  以及由 Codex、Claude、Visual Studio Code、Cursor 等客户端启动的 MCP 后台连接，并显示对应 PID。
- **不再自动结束 Agent 任务** —— 检测到占用时，安装器只提示用户保存工作并退出相关客户端，
  预检只提供“重试 / 取消”，不提供“忽略继续”，也不会代替用户结束进程。
- **防止 MCP 刚关掉又被自动拉起** —— 正式写入前再复查一次，避免客户端在第一次检查后重新
  启动 MCP，导致安装中途失败。

### 内部
- 占用检测通过 Windows WMI 只读查询进程树，同时覆盖 `python.exe` 与 `pythonw.exe`；
  完整命令行只用于内部判定，不写入日志或用户界面。全新安装没有旧运行环境时直接放行。

## [1.0.17] — 2026-07-17

扫描件现在可在深索时自动做本地 OCR；Agent 交付物与定时任务的展示和打开也更准确。

> **本版升级方式**：新增了本地 OCR / OpenCV 运行组件，已有用户请下载完整安装器，直接覆盖安装到
> 原位置即可，**不用卸载**；索引、综述、设置和 Agent 文件都会保留。若先用了应用内增量更新，
> 新版会继续提示你下载并运行完整安装器。

### 新增
- **扫描件自动本地 OCR** —— 深索 PDF 时优先读取原有文字层，只对没有文字的页面调用
  本地 OCR。识别结果只写入 PaperPiggy 索引，不修改原 PDF，也不上传文献。
- **交付物主题可直接打开** —— Agent 页的最近交付物卡片可直接跳到对应文件夹，同时正确
  统计子文件夹里的成果。

### 改进
- **区分扫描件、附件丢失和 PDF 损坏** —— 不再把所有“读不到正文”都笼统显示为“需 OCR”；
  界面会告诉用户哪些可自动 OCR，哪些需要补齐附件或更换 PDF。
- **定时任务展示更诚实** —— 成果位于子文件夹时不再误报“0 个文件”；任务目录缺少
  `任务.md` 时会明确提示未识别原因。
- **弹窗风格统一** —— 备份、恢复和应用升级不再使用会显示 `127.0.0.1:8770` 的浏览器原生弹窗。

### 内部
- OCR 新增了 RapidOCR / OpenCV 等运行时依赖；首个包含该功能的版本需通过完整安装器升级，
  旧版的“应用内增量更新”不会补齐 Python 依赖。
- 发布构建会按锁文件与实装依赖生成运行时指纹；依赖变化时强制刷新 `bundle/python` 并真实导入
  RapidOCR / OpenCV 等核心组件，避免安装器误带上一版的旧 Python 环境。

## [1.0.16] — 2026-07-17

启动和闲置时大幅少占内存：检索组件改为真正按需加载，并在空闲后自动释放。

### 改进
- **检索组件改为用到时才加载、空闲后自动释放** —— 打开应用只读取轻量的库目录；第一次在界面、对话或
  Agent 中检索时，才加载检索模型和 BM25。最后一次检索完成后默认空闲 10 分钟释放，正在进行的检索绝不会
  被中途打断。
- **释放时间可在「设置 → 检索内存」调整** —— 可选 5/10/20/30/60/120 分钟或“不自动释放”，并会显示
  当前是已释放、正在加载、正在检索，还是还剩多久释放。释放只腾出临时内存，不会删除文献、索引或设置；
  代价是释放后的第一次检索要重新准备，本地模型通常比 API 模式等待更久。

### 修复
- **不再在启动时把全部正文和 1024 维向量转成 Python 对象** —— 向量继续留在 LanceDB，检索时只按需读取
  融合后的少量候选，而且候选详情不带向量列。20 万段规模下可避免约 10 GB 的常驻内存和长时间冷启动；
  检索排序逻辑不变。

### 内部
- 建库、增量导入、AI 主题聚类等维护路径也不再无意读取整张向量列；需要遍历向量时改为紧凑的分批处理。

## [1.0.14] — 2026-07-16

后台 / 外部改动自动刷新（不用点也不用重开）+ 修「关掉再打开白屏」+ 关窗彻底关闭后端。

### 改进
- **库、综述、交付物在后台或被外部 AI 助手改动后，你正看着的那一页会自动刷新** —— 不用点刷新、更不用
  重开。外部助手写回一篇综述、或有新文献入库，当前页会在几秒内自己更新（分域感知：只刷真变了的那一域

### 修复
- **关闭应用后，后台服务会被彻底关闭**（含它启动的子进程），不再残留 python 进程越积越多。以前「关掉后

给「数据在使用中变了、却要重开应用才看得到」的地方补上刷新：库总览、综述库、浏览页各加「🔄 刷新」，Agent 页的深索进度接进后台轮询自动更新。

### 改进
- **库总览、综述库、浏览页各加了「🔄 刷新」按钮** —— 后台深索 / 生成摘要在跑，或外部 AI 助手改了库

  - 综述库：刷新综述列表 + 建议横幅（AI 助手写回综述后立刻可见）。

  轮询上）、检索 / 对话页的范围下拉（切页即重拉）本就不会滞后，无需改动，只给真正「加载一次就锁」的
  几处补了刷新。

## [1.0.12] — 2026-07-15

一条数据安全修复（Zotero 库临时读不到时不再误清空索引）+ Agent 页两处体验改进（刷新按钮、交付物列表美化）。

### 修复
- **Zotero 数据目录临时读不到时，不再静默清空整个索引**（重要，关乎数据安全）。
  以前：你在设置里指定了 Zotero 库的位置（比如放在 D 盘），若某次「更新知识库」时那个盘 / 目录
  恰好读不到（盘没插好、目录被改名），程序会**悄悄回退**到一个早已废弃的空 Zotero 库，把它当成
  「你的库现在几乎没文献了」，进而把已建好的整库索引（含深索成果）**当作"已删除"清空** —— 全程不报错。
  现在两道防护：① 你**显式指定过**的库若当前读不到，直接**停下报错**（"读不到你配置的 Zotero 库…"），
  绝不改用别的库；② 即使别的原因让读到的文献数骤减，清理前也会**先过一道阈值**（读到空库、或一次要删
  过半 → 拒绝清理并提示），宁可暂留几条残留，也不误抹整库。

### 改进
- **Agent 页「⏰ 定时任务」和「📦 最近的交付物主题」各加了「🔄 刷新」按钮** —— 新建的定时任务、
  新写好的交付物，**点一下刷新就能看到，不用再重启应用**（此前这两块只在应用启动时加载一次）。
- **「最近的交付物主题」列表改成卡片式**：每个主题一张卡（名称 + 日期 · 文件数 · 含说明），长名字
  自动换行、不再被截断，深浅色都适配；空的时候也会显示（附刷新提示），不再"有了交付物却看不到"。
- **法学检索分词词典补充了一批术语**（如「刑事附带民事公益诉讼」「性侵害未成年人」「量刑折扣」等），
  相关主题检索时更容易整词命中。

## [1.0.11] — 2026-07-15

修「深索显示永远卡在 99%」——其实早就深索完了，只是那几篇扫描件被算成了"待深索"。

### 修复
- **首页「① 深索」卡（以及深索面板、Agent 页、深索邀请）把扫描件也算进"待深索"，导致永远显示
  「99% · 还有 N 篇」**，让人误以为卡住或没完成。实际上那 N 篇是**扫描件/无正文**（没有文字层、
  需 OCR，本来就无法深索）。现在进度按"**已处理**"算（已深索 + 已确认扫描件）：全部有正文的都深索完、
  剩下的都是扫描件时，显示 **100% · 已全部深索完成 ✓**，并注明"另有 N 篇扫描件无法深索"，
  不再有误导性的"还有 N 篇 / 深索全部未深索文献"。真正还有可深索文献时，照旧正常提示。
  （1.0.9 修过顶栏进度条与深索详情面板的同类问题，这次补齐首页与其余入口。）

顶栏自动提示新版本；并为「朋友在 Mac 上从源码跑」做了跨平台适配（Windows 行为不变）。

### 新增
- **右上角自动更新提示**：有新版本时，顶栏右上角（⚙ 设置 左边）会出现「🎁 有新版 vX →」，
  点一下直接进设置的更新区一键升级。启动时会静默检查一次（设置 → 应用更新 里有「启动时自动检查
  新版本」开关，**默认开**，不想让它每次启动连 GitHub 就关掉）。右键徽标可忽略当前版本，
  发更新的版本时会自动重新提醒。
- **macOS 从源码运行支持**（面向个人自用，非公开发布的打包）：数据落点改到
  `~/Library/Application Support/PaperPiggy`、依赖按平台分流（macOS 装 pyobjc、跳过 Windows 专属的
  pythonnet）、Zotero 目录探测支持 macOS profile、打开文件夹/取消建库等改为跨平台。
  详见 [MAC-从源码运行.md](MAC-从源码运行.md)。**以上改动在 Windows 上一律无变化。**

## [1.0.9] — 2026-07-15

为「重装一次库、深索重来一遍」铺路：只导 PDF 更严格、深索不再假装卡在 99%、并新增一键清空重建。

### 修复
- **深索到 99% 就「卡住不动」（其实没卡）**：所有有正文的文献都嵌完之后，还有最后一步——
  重建 bm25 检索索引（几十万个文本块，分词+建索引要好几分钟）。这期间进度条冻在 99%、
  深索计数也不动，看着像死机，其实在收尾。现在这一步会明确显示「**深索已完成，正在重建检索索引（收尾中）**」，
  不再误导你以为卡死。
- **「只导入有 PDF 的文献」以前会放行法源类**：法规/案例/标准/报告即使没有 PDF 也照样进库，
  和开关字面意思不符、也让一堆无正文的纯题录混进来。现在**严格只导有 PDF 的**——
  重要法条你自己配上 PDF 再入库即可。

### 新增
- **一键「清空并从头重建索引」**（设置 → 🧹 清空并从头重建索引）：想把索引推倒重来时用。
  它**只清索引**（全文提取、切块、向量、bm25、深索标记、整库状态），
  **明确保留**综述库、收藏夹分类、期刊分级、检索摘要（花过 API 钱的）、页码映射、设置、
  `0_Agent` 工作区。清空后点「⟳ 更新知识库」重建题录+语义（会**自动去重、按“只导 PDF”重新导入**），
  再「深索全部」重新深索。旧的提取/切块会移到一个备份目录、可手动捞回。
  ⚠️ 向量索引会被丢弃、需重新嵌入，操作前建议先在「💾 备份与恢复」打一份含索引的备份。

## [1.0.7] — 2026-07-15

自愈 1.0.6 之前那次崩溃留下的「僵尸扫描件标记」——1.0.6 修好了 PDF 提取，但之前被误判的

## [1.0.6] — 2026-07-15

修复深索时应用崩溃 + 大量文献被误判为「扫描件」——根因是 PDF 提取库的线程不安全。

### 修复
- **深索时整个应用窗口崩退**：PDF 提取用 4 线程并发调 PDFium，而 **PDFium 不是线程安全的**，
  多线程同时访问会损坏其内部堆（Windows 事件日志确诊：故障模块 `pdfium.dll`，
  异常码 `0xc0000374` 堆损坏 / `0x80000003`）。改为**所有 PDF 提取全局串行**，绝不并发。
  实测：同一批 500 个 PDF，并发时约 70% 失败/偶发崩溃，串行后 100% 成功、不崩。
- **大量有文字的文献被误标「🚫 扫描件·需 OCR」**：正是上面并发竞态导致的随机
  「Failed to load」（同一 PDF 单独打开明明好好的），被误判成扫描件、深索篇数虚低。
  串行修复后，重跑「深索全部」会自动重试这些篇并成功入库。
- PDF 提取加瞬时失败重试（文件被 OneDrive / 杀软 / Zotero 短暂占用时多试两次）。

> 升级后请重新点一次「深索全部」：之前因崩溃/误判没入库的文献会自动补上（不会重复劳动，已成功的跳过）。

## [1.0.5] — 2026-07-15

修复 API 模式下批量深索/建索引的崩溃、假完成与卡死。

### 修复
- **API 模式深索全库会崩**：嵌入调用没有兜底，一次 API 失败（余额不足→403、限流、网络超时）
  就会带 traceback 掀翻整个后台任务。现在：致命错（密钥/余额/模型名）立即优雅中止并给人话提示；
  限流/网络类跳过该篇、连续多次才判停；**任一篇没嵌成功都不会谎报完成**，已嵌的部分照常保存、可续跑。
- **崩了却提示「✓ 深索完成」**：失败后紧接着的后台任务会把全局退出码覆写掉，前端据此误报成功。
  现在后端失败时不再立即接排新任务、前端只认「退出码 0」的阳性证据才报完成，否则一律「⚠ 深索中断」。
- **进度条卡死在「正在提升检索质量 X/Y」**：语义嵌入中途失败后，进度条会永远停在那儿谎报进行中。
  现在区分「真在跑」（动进度条）与「上次未完成」（静态提示「点『手动更新知识库』继续」）。
