# 实现设计 · 半自动研究助手（选题·框架·带页级引注的资料汇编）

> **本文件是什么**：把《LocalKB · Agent 工作流蓝图》波次一的旗舰「写论文工作流」的**前半段**收敛成一个可独立交付的产品——一个「半自动研究助手」。它帮中文法学学者做三件事：①选题/标题/大纲的启发；②基于知识库的可用材料梳理（**引注到期刊印刷页码级**）；③建议新增文献（用期刊分级系统排序）。目标是让一个全新对话读完本文即可动手实现，每个 Phase 独立交付、独立验收。
> **怎么用**：新对话里说「读 `docs/设计/实现设计_半自动研究助手.md`，从 Phase A 开始实现」。
> **对标产物**：开发者用旧知识库临时生成过一份同类的「资料汇编」docx，**它就是本助手要产出的东西的样板**（结构、引注形态、覆盖评估、补检索建议、学术风险提示均照它）。**关键升级点：旧版引注的"第X页"是 PDF 顺序页号；本设计要求一切引注落到"PDF 内部实际印着的期刊印刷页码"**（读者翻期刊看到的那一页）——这是本设计的技术核心（§3）。
> **依据**：对源码全量核对（file:line 在正文）+ 对 1444 份真实中文法学 PDF 逐页文本的**页码样式实证**（§3.1）+ 《法学引注手册》(2019/2版，何海波) 引注体例 + 《LocalKB · Agent 工作流蓝图》的三条红线与三条铁律。
> **定位纪律**：半自动**研究助手**，不是自动成文机。每一步产物都是"带源初稿 + 待学者核校"，学者对内容负全责（见 §7）。
> **日期**：2026-07-08

**把"检索器"升格成"会替你把库里可用材料按你的选题梳理成带页级引注的资料汇编、并诚实告诉你哪里缺料、该补哪些文献"的研究助手。**

1. **选题/标题/大纲启发**（§5.1）——把模糊方向拆成"方法论视角/研究对象/制度落点"式关键词、问题意识、核心命题、创新点、标题候选、grounded 三级大纲。

3. **建议新增文献**（§5.3）——用期刊分级 + 引文挖掘 + 覆盖缺口，给出"该补哪些文献/该补检索什么"，补入后可迭代重跑（对标 docx 里"作者最新补入 5 篇核心文献"的闭环）。

## 3. ★核心技术：PDF 顺序页 → 期刊印刷页码（`page_map`）

这是本设计相对旧版的关键升级，也是"引注到 PDF 内部期刊页码"的落地基础。**必须先做（Phase A），它是能力二一切引注的地基。**

### 3.1 实证：中文法学 PDF 的印刷页码长什么样（来自 1444 份真实提取）
- **印刷页码稳定出现在每页文本的尾部**，绝大多数是**间隔号包裹的数字**：
  - `·53·` `·54·` `·55·`（《法学研究》，半角）
  - `·145·` `·146·` `·147·`（《中国社会科学》）
  - `·４１９·` `·４２０·` `·４２１·`（《中外法学》，**全角数字**）
- 少数期刊是页尾**裸数字**：`…第155页 。 5`（《政治与法律》，尾部孤立 `5`）。
- **页眉**是"《刊名》YYYY年第N期"（偶数页）/文章标题（奇数页）交替，如 `法学研究 2023 年第 2 期`、`中外法学 ２０２０ 年第 ２ 期`——**可顺带解析出年/期**，供生成完整手册脚注（§4）。
- **关键区分**：正文/脚注里的 `第X页`（如"参见…第 8 页"）是**脚注 pinpoint**，不是本页印刷页码；**页尾的 `·NN·` 才是印刷页码**——两者形态不同、可靠区分。
- **连续性**：印刷页码随 PDF 页序 +1 连续，一个 PDF 页 = 一个印刷页。配合 LocalKB `extract.py` 已写入 extracted meta 的 `official_pages`（整篇页码范围，如"45-67"）可做**强锚定**。

### 3.2 检测算法（`page_map.build(stem)`）
对每篇 `data/extracted/<stem>.json` 的 `pages:[{page, text}]`：

1. **抽候选**：对每页，取**尾部 ~60 字**（次选头部）扫描，正则（先全角→半角归一）：
   - 主：`[·•・‧．]\s*([0-9]{1,4})\s*[·•・‧．]?\s*$`（间隔号包裹，页尾）
   - 备：`(?<![0-9第])([0-9]{1,4})\s*$`（页尾裸数字，排除紧跟"第"的 pinpoint）
   - 全角数字 `０-９` 先 `str.translate` 成半角再匹配。
2. **锚定**：设 `official_pages="45-67"` → 值域 `[45,67]`、长度 23。在各页候选里找**最长的、随 PDF 页序单调 +1、且落在值域内**的整数运行段（run）。该 run 唯一确定 `pdf_page → printed_page` 的若干锚点（滤掉正文里的年份/条号等噪声数字）。
3. **补全**：未检出的 PDF 页用锚点按步长 1 内插/外推（因连续），`method="interp"`。
4. **兜底**：若整篇检出失败（扫描件/无文本层），用偏移法——`printed = start + (pdf_page - first_content_pdf_page)`，`first_content_pdf_page` 取第一页（或跳过检出的封面/目录页），`method="offset"`，`quality="low"`。
5. **无 official_pages**：仅靠 §3.1 的 `·NN·` 检出运行段也能建映射（多数中文刊可直接检出），只是少了值域校验，`quality` 降一档。

### 3.3 sidecar 与解析接口
- 存 `data/pagemap/<stem>.json`（与 extracted 并列，只写 DATA，随便删可重建）：
```jsonc
{ "stem": "...", "key": "...", "official_pages": "45-67",
  "issue": "2023年第2期",           // 从页眉解析（可空），供完整脚注
  "quality": "high|med|low",
  "map": { "1": {"printed": 45, "method": "detected", "conf": 0.98},
           "2": {"printed": 46, "method": "interp",  "conf": 0.8}, ... } }
```
- 解析函数 `page_map.printed(key_or_stem, pdf_page) -> {printed:int|None, method, conf, display}`；`display` 为引注用字符串：高置信 `"52"`；低置信/推算 `"52(页码推算)"`，让不确定**显式暴露**（三态诚实，蓝图铁律三）。
- **构建时机**：随 deep-index（`extract.py` 之后）批量建；或按需对命中篇 lazy 建 + 缓存。断点续跑（已存跳过），仿 `sac.py:62 ensure_for` 幂等模式。

### 3.4 接进检索/引注
- 检索命中的 chunk 已带 `page`（= PDF 顺序页，见 `chunk.py:72` `chunk_id=f"{key}::p{page}::c{ci}"`、`search_full` 输出 `"page": r.get("page")`）。
- **引注解析**：`page_map.printed(key, chunk.page).display` → 印刷页码，喂给 `cite_format`（§4）。
- 改 `retriever._page_cite`（retriever.py:166）：全文块引用的"第{pg}页"改用 `page_map.printed` 的印刷页码（当前 `pg = official_pages or page` 是整篇范围/PDF 序，不合要求）；wiki 行分支不变。**这一步同时让网页 `/chat`、检索卡、资料汇编三处引注全部升级到印刷页码。**

> **为什么可靠**：`·NN·` 页尾标记 + `official_pages` 值域 + 连续性三重约束，互相校验；检出失败也有偏移兜底且显式标"推算"。这比旧版"直接用 PDF 序号"精确一个数量级，且诚实标注不确定项。

## 4. 引注格式引擎 `cite_format.py`（《法学引注手册》子集）

**规则做格式、模型做实质**（蓝图铁律二）：格式确定性部分用规则，绝不交 LLM。两种输出形态：

1. **资料汇编紧凑式**（对标 docx，第二部分正文用）：
   `[{作者}《{题名}》，{刊名}，第{印刷页}页]`
   - 作者取 `author` 首作者（多作者可加"等"）；题名去副标题冗余；刊名用 `journal`；页码用 §3 印刷页码。
2. **完整手册脚注式**（生成初稿/导出时用，对齐《法学引注手册》页下脚注）：
   `{作者}：《{题名}》，载《{刊名}》{年}年第{期}期，第{印刷页}页。`
   - `年` 取 meta `year`；`期` 取 §3.3 从页眉解析的 `issue`（无则留空并标"待补期号"）；著作/文集/报纸/网络/案例/法规各套模板（itemtype 驱动）；〔〕vs（）、"载"、书名号规则化。

**必须人判、不自动改**（铁律二）：引领词 `参见/见/又见/另见/转引自` 的选择——它是对"作者是否核实原文"的实质声明。**本助手默认不加引领词**（紧凑式直接 `[作者《题名》…]`），或标注"（引领词待定：本条为直接命中，若作概括引用请人工改为『参见』）"；**转引场景**（§7 归属风险）由助手**提示**"疑似转引自 X，请核原文后决定标『转引自』"，不擅自改写。

## 5. 模块与数据流

### 5.1 能力一：选题 / 标题 / 大纲启发
**输入**：用户给研究主题/方向（+ 可选：指定 Zotero 收藏夹路径或 AI 主题簇 id 作为范围）。
**流水线**：
1. **范围映射**：对主题跑 `R.search`（topk 调大、`sort="blend"`）+ 命中 `ai_topics.json` 的簇 + `zotero_collections.json` 的收藏夹 → 勾勒库内覆盖面（哪些簇/收藏夹相关、各多少篇、tier 分布）。
2. **选题拆解**（LLM）：把主题拆成"方法论视角/研究对象/制度落点"式关键词结构（对标 docx Part1 一）；产出**问题意识**候选、**核心命题**（可支撑方向）、**可能的创新点**（每条挂支撑文献 key，只用召回到的真实篇）。
3. **标题参考**（LLM）：生成若干主/副标题候选，对齐主题与核心命题。
4. **三级大纲**（LLM，grounded）：基于已召回证据生成大纲，**每个大纲节点挂 ★核心/☆辅助 文献 key**（★ 优先高 `journal_weight` 篇，用 `journal_grading`）。
   **★人工闸**：用户确认研究问题、改大纲——这是全流程最硬的闸，学者的论证主线必须自己定（蓝图 §7 闸1）。
**产物**：第一部分 一/二。持久化为 wiki 页（新 `kind=outline`，G3）。
**复用**：`R.search`、`_load_ai_topics`(server.py:352)、`_load_cats`(server.py:330)、`journal_grading.resolve_journal_weight`、`L.chat_once`/MCP、`save_answer`。

### 5.2 能力二：页级引注的材料梳理（主体）
**输入**：能力一确认的大纲（每个叶子节点 = 一个子题 query）。
**对每个大纲节点循环**：
- **A 召回**：`R.search(节点query, topk 较大如 12-20, sort="blend")` → 命中 chunk（带 `key/page(PDF序)/journal/journal_tier/text/parent_text/official_pages`）。可 2-3 轮迭代补召（借 Undermind 思路：抽高频词/相邻概念再检），设轮数硬预算。
- **B 页码解析**：对每个命中 chunk，`page_map.printed(key, chunk.page)` → 印刷页码（§3）。
- **C 综述合成（RCS）**：LLM 把该节点命中证据合成综述 prose——**每段先压成"论点 + 出处(key+印刷页)"的带出处 summary 再串写**（PaperQA2 的 RCS）；**只允许引用召回到的真实 chunk，禁止无出处生成句**（grounded）；每个论点后接 `cite_format` 紧凑式引注；同一论点多处出处连续 `[...][...]`。
- **D 缺口标注**：判断本节召回是否稀薄/偏移主题（命中篇数、tier 分布、与节点 query 的 rerank 分布）→ 生成"（资料缺口：本节命中多围绕 X，直接论述 Y 的段落较少，仅 A/B/C 三篇较直接，建议就 Y 补检索。）"（对标 docx 每节末缺口提示）。
- **E 覆盖评级**：该子题 → ◎充分/○部分/△薄弱/▲基本缺失/▽零星（按命中密度 + 高 tier 命中数）。
  **★人工闸**：每节草稿人工审校，重点核**观点归属**（谁主张什么，不混为通说）、**转引**（疑似转引须标）、**争议双呈**（§7）。
- **（可选）核验闸**：接蓝图 G1——每条引注回查 `data/extracted` 该印刷页文本做 quote/命题一致核验，不过则红标"待核"。
**产物**：第二部分全部 + 第一部分 三（覆盖评估表）。持久化 `kind=digest` wiki 页；可导出 docx（复用系统 python-docx，仿本项目 md→docx）。
**复用**：`R.search`、`data/extracted`、`page_map`(新)、`cite_format`(新)、`journal_grading`、`synthesize_topic` 深化、`_gather_evidence`（剔旧 wiki 行防自证）、`save_answer`。

### 5.3 能力三：建议新增文献（期刊分级）
三条子路径，产出第一部分 三（缺口）+ 四（补检索建议）：

- **(a) 引文挖掘（最有据）**：从本主题命中文献的 `data/extracted` **脚注文本**里抽被引文献（正则捕 `参见?/见/转引自 {作者}：?《{题名}》，?载?《{刊名}》…第X页` 等模式，复杂的交 LLM 解析）→ 归一去重 → 与 `papers.jsonl` 现有 `key`/题名/作者比对 → **库内缺失**的，按 `被引频次 × 引用者 journal_weight` 排序；对解析出的刊名调 `journal_grading.resolve_journal_weight` **估算缺失文献层级**（`needsReview` 标待核）。输出："建议新增：{作者}《{题名}》（疑似载《{刊名}》，估算 {tier}），被库内 {n} 篇引用（如 A《…》第X页、B《…》第Y页）"。
- **(b) 覆盖缺口关键词**（对标 docx Part1 四(一)）：对覆盖评级 △▲▽ 的子题，LLM 生成**分优先级、分区**的补检索关键词清单，每条说明"填补哪个缺口"（如"比较法一手与实证 / 刑事一体化→未成年人专属对接 / 理论溯源"）。
- **(c) 库内错配**：扫 `papers.jsonl` 里 `has_pdf=true` 但未深索（不在 `embedded_keys.txt`）、或高 `journal_weight` 但不在本主题收藏夹/簇的相关篇 → "已有但未纳入本主题深索，建议深索/纳入"。
**期刊分级贯穿**：排序优先高 tier 缺失；覆盖评估显示各子题命中的 tier 分布；估算层级 `needsReview` 显式标注。
**★人工闸**：建议清单人工勾选 → 用户决定补哪些进 Zotero → `localkb_build(stage=deep)` 补索 → **重跑能力二**（迭代闭环，正是对标 docx"作者最新补入 5 篇核心文献"的来历）。
**复用**：`data/extracted`(脚注)、`papers.jsonl`、`journal_grading`、`R.search`、`ai_topics`/`collections`、`embedded_keys.txt`。

## 7. 诚信红线守法（直接提炼自对标 docx 的"学术风险提示"）

对标 docx 的第一部分 四(二) 已经把中文法学写作的诚信雷区列全了。本助手**必须把这些做成产物里的内建提示**，而非靠用户自觉：

1. **观点归属风险（最需警惕）**：凡涉原创者观点但库内只有转引命中（如储槐植"刑事一体化"命题多为他人转引发挥），须标"转引自×××"或提示"补原著后直引"，**不可将转引者的发挥径直归于原作者**。→ `cite_format` 转引提示 + G1 归属核验；对标 docx 第1节明确写了这条缺口。
2. **立场归属不可混为通说**："实体出罪"（黎宏、李程林）vs"程序出罪"（王迎龙、刘仁文、张杰一侧）须清晰标立场归属，**不可写成"通说"**。→ 防 false attribution（蓝图 4.6 最核心失败模式）。
3. **争议双呈**：真实学界争议须"先陈两方、再表本文立场"，避免单边裁剪（对标 docx 列了四处：实体vs程序出罪 / 递进说vs并列说 / 专门教育vs专门矫治 / 认罪认罚"从宽"是否等于"从快从简"）。→ 助手在综述里以"分歧提示"块并列双方 + 各自页码，**绝不落成"A 错 B 对"断言**（蓝图铁律：矛盾不当断言）。
4. **数据时效**：转引数据标原始年份/出处，跨年比较注明各来源口径可能不一（对标 docx 提醒了附条件不起诉率、酌定不起诉率序列等）。
5. **比较法转引强度控制**：域外材料以中文转引为主时，**不以"德国法规定"口吻断言**，降格为"据×××介绍"；涉具体法条建议补一手译本。→ 助手对 `langid` 非中文原典/仅中文转引的证据自动降调并提示。
6. **缺口诚实（三态铁律）**：每节标"资料缺口"，"库内没有 ≠ 学界空白"；缺口措辞用"在本库 N 篇覆盖范围内、截至 DATE 未见充分讨论"，禁绝"首次/学界空白/无人研究"。
7. **AI 披露**：产物内置《生成式 AI 使用声明》草稿（工具名+版本、参与阶段=检索/材料梳理/大纲启发、贡献度、核验方式）——对接《学位法》§37、CAS《指南 2.0》、复旦/中科院规范（蓝图红线2）；助手只做"带源初稿 + 核验助手"，终稿由学者改写（抗 CNKI 检测，红线3）。
8. **判例/法条不由助手生成**（红线1）：本助手只对库内期刊论文做综述；任何判例、案号、司法解释一律学者手工补或走权威库另核，助手不凭生成产出。

## 8. 分期路线（每 Phase 独立交付 + 验收）

### Phase A — 印刷页码映射 `page_map`（地基，最高杠杆，先做）
- 建 `page_map.py`（§3 检测算法 + sidecar + `printed()` 解析）；随 deep-index 批量建 / 按需 lazy 建。
- 改 `retriever._page_cite`(retriever.py:166) 用印刷页码；`GET /research/pagemap/{key}` 调试端点。
- **验收**：对 20 篇不同期刊（含全角数字刊如《中外法学》、裸数字刊如《政治与法律》）抽查，`printed(key, pdf_page)` 与人翻 PDF 对应页一致；检出失败篇 quality=low 且引注显式标"页码推算"；网页 `/chat` 引注同步升级为印刷页码。**这一步单独就把"引注到期刊印刷页码"这个你最强调的需求兑现了。**

### Phase B — 能力二 digest（单节 → 整份）
- `cite_format.py`（§4 紧凑式）+ `POST /research/digest`：给一个 query → 召回 → page_map → RCS 综述（带印刷页引注）→ 缺口标注 → 覆盖评级；存 `kind=digest` wiki 页。
- **验收**：给一个子题得到一节带 `[作者《题名》，期刊，第印刷页页]` 的综述，随机 5 条引注翻 PDF 命中该页该论点；节末有诚实的资料缺口提示。

### Phase C — 能力一 outline + 组装整份汇编
- `POST /research/scope`：主题 → 范围映射 + 选题拆解 + 标题候选 + 三级大纲（★/☆）；`kind=outline` 页。
- 组装：大纲各叶子节点跑 Phase B → 拼成完整《选题·框架·资料汇编》；`POST /research/export_docx` 导出对标 docx。
- **验收**：给一个主题，产出结构与对标 docx 一致的完整文档（两部分齐全，覆盖评估表在位）。

### Phase D — 能力三 建议新增文献 + 迭代闭环
- `POST /research/suggest_sources`：引文挖掘（脚注解析缺失文献）+ 覆盖缺口关键词 + 库内错配；按 tier 排序。
- 迭代：勾选 → 补 Zotero → `localkb_build(deep)` → 重跑 Phase B/C（标"最新补入"）。
- **验收**：给出按期刊层级排序的"建议新增 N 篇 + 补检索关键词清单"，每条附"被库内哪些篇引用/填补哪个缺口"；补入 2 篇后重跑，汇编相应章节变厚并标"最新补入"。

### Phase E（可选）— 接 G1 核验闸 + 手册完整脚注
- 每条引注 G1 页级 quote/命题一致核验；`cite_format` 完整手册脚注式（含期号、引领词人工确认）。

**最高杠杆一步 = Phase A**：它把你最强调的"引注到 PDF 内部期刊页码"单独兑现，且是能力二一切引注的地基；近确定性、可独立验收、风险最低。

## 9. token 成本

- **Phase A page_map**：纯规则/正则，**零 LLM**（仅解析失败的复杂脚注可选 LLM 兜底，极少）。
- **能力二每节 digest**：召回(本地免费) + RCS 逐证据压缩(每证据 ~1k) + 合成 ≈ 每节 2 万–5 万 token；一份 6 节汇编 ≈ 15 万–30 万 token。缓存命中/未改节 0 成本。
- **能力一 outline**：范围映射(免费) + 拆解/大纲 1-3 次 ≈ 3 万–8 万 token。
- **能力三 suggest**：脚注解析（规则为主，复杂交 LLM）+ 缺口/关键词生成 ≈ 5 万–15 万 token/次。
- **分层**：page_map/引文挖掘规则化的部分免费；综述合成用前沿模型（经 MCP）保质量或廉价 key 跑量。**never eager**：每次 digest 对应一次用户显式请求。
