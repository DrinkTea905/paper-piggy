## 0. 一句话与核心决策

**产品的生死线是"上来就能用"**：新用户下载打开后，不能等数小时全文建库才第一次能检索。解决方案 = **一条索引、三档渐进、按需深索**：

| 档 | 名字 | 索引什么 | 耗时 | 触发 | 给用户什么 |
|---|---|---|---|---|---|
| **L** | 即时词法层 | 全库每篇「标题+摘要+关键词」，jieba/BM25 词法 | **秒级**（0 嵌入） | 连上 Zotero 立刻自动 | 打开就能按主题/作者/刊名搜到篇目；仪表盘立刻点亮 |
| **S** | 快速语义层 | 同上，但 bge-m3 向量（约 2000 条短文本） | **1–2 分钟** | L 之后自动后台 | 近义/语义召回；检索质量提升 |
| **F** | 全文精读层 | 全文提取→切块→约 19 万块向量+reranker | **数小时** | **opt-in / 后台 / 按需** | 页码级、论点级 grounding（现有管线） |

**关键实现原则（来自对抗评审，务必遵守）：**
1. **不建第二张 LanceDB 表、不给用户"浅搜/深搜"模式开关。** 用一列 `row_type` 区分「题录行(meta)」和「全文块行(chunk)」，同一张表、单一查询。结果随后台完成"自动变厚"，卡片只标一个小徽章（`仅摘要` / `📄 全文可精读`）。
2. **"要好久的嵌入"= 只有 F 档。** L 档 0 嵌入（纯词法），S 档只嵌约 2000 条短文本。用户的等待焦虑靠"L 秒级可用 + 进度条"化解，而不是靠让他等 F。
3. **F 档默认不自动跑全库**，由用户决定（全库 / 某收藏夹 / 单篇按需）。

## 1. 现状快照（改之前必须知道的事实，均已核对源码）

### 1.1 现有建库管线（`build_all.py` 串行编排三步）
- **`extract.py`**：解析 `.bib` → 每篇 PDF 逐页文本写 `data/extracted/<safe_key>.json`。
  - `meta_of(e)`（L50-62）只保留 `key/title/author/year/journal/doi/langid/keywords/abstract/itemtype`，**丢弃了 `pages`(官方页码范围)/volume/number/issn/publisher 等**。
  - `pdf_paths(e)`（L64-67）已修好 Better BibTeX 的 `\:`/`\\` 反转义。**无 PDF 的条目 `if not paths: continue`（L130）直接跳过**——所以纯题录/网页文献目前在系统里完全不可见。
  - 用 **ThreadPoolExecutor**（不是多进程——避开本机 venv 瘦启动器 + Windows spawn 把 `sys.executable` 解析成系统 Python 的坑，见 [坑#1]）。
  - `safe_name(key)`（L27-31）把 Windows 非法字符 `<>:"/\|?*` 换成 `_`，超 200 字符加 md5。**diff key 时必须用 `safe_name` 归一化**，否则已入库误判为未入库。
- **`chunk.py`**：`data/extracted/*.json` → `data/chunks/<safe_key>.json`（子块数组）。
  - 每块字段：`chunk_id`(=`{key}::p{page}::c{ci}`) / `key` / `page` / `heading` / `text`(子块~500字) / `parent_text`(以子块为中心的≤2400字页窗，L44-52 已修错位) / `title/author/year/journal/doi/langid`。**注意：块里没有 journal_tier/abstract/keywords/itemtype。**
- **`embed_index.py`**：切块 → bge-m3 向量 → LanceDB 表 `chunks`；jieba → bm25s 词法索引。
  - **表 schema（当前 15 列）**：chunk 的 13 个字段 + `vector`(fixed_size_list<float>[1024]) + `journal_tier`(嵌入时用 `JT.tier_of(journal)` 现算写入)。
  - `journal_tier` 写入条件 `want_tier = (tbl is None) or ("journal_tier" in tbl.schema.names)`（L54）。
  - 进度账本 `data/state/embedded_keys.txt`（每行一个 safe_key stem）。
  - `add` 前按原始 key 先 `delete`（幂等，防休眠杀进程导致重复；`dbutil.key_predicate`）。
  - 无新增且 bm25 覆盖全表 → 秒退。SAC 摘要前缀（`data/summaries/summaries.json`，**LocalKB 当前为空**）。

### 1.2 检索与服务
- **`retriever.py`**：`load_all()` 把整表读进内存 `M["records"]`(chunk_id→row) + bm25。`search(query, topk, sort)`：dense(bge-m3/LanceDB cosine, top50) + bm25(top50) → RRF 融合 → 候选池 `max(50, topk*8)` → reranker 精排 → **按 key 去重**(`MAX_PER_KEY=2`) → 三种排序 `relevance|tier|blend`(默认 blend)。
  - **输出字段**：`chunk_id/score/journal_tier/tier_rank/title/author/year/journal/doi/page/key/heading/text/context(=parent_text)/citation`。
  - `citation(r)`（L66-74）拼「作者等(年)《标题》，刊名，第{page}页」——**这里的 `page` 是 PDF 内页码，不是官方期刊页码**（违反用户长期规则，见 [改动] 官方页码）。
  - ⚠️ `retriever.py` 有自己的 FastAPI app + idle_watch，但**在产品里是被 `server.py` in-process import 的**（`import retriever as R`；调 `R.load_all/search/STATE/M`）。standalone 模式(`python retriever.py`)是备用，产品不用。
- **`server.py`**：FastAPI @ `127.0.0.1:8770`，一个进程全包。现有路由：
  - `GET /` → `web/index.html`；`/static` 挂载 `web/`。
  - `GET /health` → `{ready, n_chunks(=内存块数), building}`。**这是唯一的"状态"接口，只有一个标量，喂不了仪表盘。**
  - `POST /search` `{query, topk=8, sort?}` → `{query, took_ms, results[]}`；未就绪返回 503。
  - `POST /chat` `{query, history, provider, base_url, api_key, model, topk=6, sort?}` → SSE 流；先推 `{sources:[...]}` 再推 `{delta:...}`，末 `[DONE]`。**用户自备 LLM key，不存服务端。**
  - `POST /build` → 起子进程跑 `build_all.py`，完成后 `R.load_all()` 热重载；`BUILD={running,log,started}` 内存态。
  - `GET /build/status` → `{running, log[-50:]}`（内存日志，重启即失，CLI/计划任务跑的建库它看不到）。
- **`config.py`**：所有路径/参数集中处。`MODELS`/`BIB` 可用环境变量 `LOCALKB_MODELS`/`LOCALKB_BIB` 覆盖（**分发的关键**）。端口 8770，`EMBED_DIM=1024`，`MAX_PER_KEY=2`，`TABLE_NAME="chunks"`，`DEFAULT_SORT="blend"`，`TIER_BONUS` 加权表。
- **`web/`**：零依赖 vanilla JS，两个 tab（🔍检索 / 💬对话）+ 两个 modal（设置/建库）。深板岩顶栏 `#1e293b`，强调蓝 `#2563eb`，`TIER_COLOR` 7 色表(0红→6灰)。tab 切换在 `app.js` L40-46，**硬编码只切两个 panel，加第三个 tab 要把它泛化(约3行)**。无图表库、无 CDN（离线）→ 图表**手绘 SVG/CSS**。

### 1.3 已建好、待接线的能力（近乎免费的赢）
- **期刊 tier 排序**：`retriever.search` 已支持 `sort=tier/blend/relevance`，每条结果已带 `journal_tier`+`tier_rank`；**检索页 UI 没放排序控件，还把 author/year/page 丢了没显示**。接上即得。
- **摘要/关键词/类型**：`.bib` 里 `meta_of` 已解析出来，只是没往下游传。做 L 档/仪表盘/分类直接捡起来用。
- **官方页码**：`.bib` 有 `pages` 字段（如 `112-130`），`meta_of` 丢了。修它成本极低、且是用户硬规则。

### 1.4 已知坑（务必别重踩）
1. **venv 瘦启动器 + spawn**：`ProcessPoolExecutor` 在本机把 `sys.executable` 解析成没装包的系统 Python312 → 崩。**一律用 ThreadPoolExecutor**（pymupdf 释放 GIL 照样并行）。
2. **`.bat` 编码**：Windows cmd 按 GBK 读 `.bat`，含中文路径的必须存 **GBK**，UTF-8 会"找不到路径"。
3. **`PYTHONUTF8=1` 传染 subprocess** → 逼 pymupdf 内部 subprocess 用 UTF-8 解 GBK warning 而崩。`build_all` 特意 `pop` 掉它，子进程输出 `decode(errors=replace)` 兜底。（注意：`server.py` 的 `/build` 现在**设了** `PYTHONUTF8="1"`——迁移时留意一致性。）
4. **key 归一化**：`embedded_keys.txt` 存 `safe_name` stem；任何"已入库 vs 未入库"差集都要先 `safe_name` 归一化。删行用 `dbutil.key_predicate`（转义单引号构造谓词）。
5. **本机休眠杀进程树**：长任务(F 档全库)过夜/合盖会被杀。用自愈计划任务（见 memory `long-task-sleep-resilience`）+ 断点续 + 文件锁。

## 2. 目标数据模型：统一表 + `row_type`

LocalKB **全库尚未建**（现仅约 20 篇测试库），且产品在每个用户机器上都是**全新建库**——所以**不需要迁移**，直接用新 schema 建表。

### 2.1 统一表 `chunks`（新 schema）
在现有 15 列基础上，**新增 5 列**（所有行都带，便于仪表盘/分类/检索直接查表，不必回读 extracted JSON）：

| 列 | 类型 | meta 行 | chunk 行 | 用途 |
|---|---|---|---|---|
| `row_type` | str | `"meta"` | `"chunk"` | **区分档位**；检索/统计据此分流 |
| `itemtype` | str | article/book/report… | 同篇继承 | 类型分面 |
| `official_pages` | str | `.bib` pages(如"112-130") | 同篇继承 | **官方页码引用** |
| `has_pdf` | bool | 有无 PDF | true | 覆盖统计 / 能否深索 |
| `ingested_at` | str(ISO) | 写入时间 | 写入时间 | "最近加入"/增长时间线 |

- **meta 行**：一篇一行。`chunk_id = "{key}::meta"`；`page=null`；`text = title + "\n" + abstract + "\n" + keywords`（去 emoji；这既是展示也是嵌入单元）；`parent_text = abstract`；`vector` = S 档嵌入 `text` 得到。其余 title/author/year/journal/journal_tier/doi/langid/itemtype/official_pages/has_pdf 照填。
- **chunk 行**：与现有完全一致，只是多带 `row_type="chunk"` + 4 个新列（同篇 meta 复制）。
- **向量同空间**：meta 与 chunk 向量都是 bge-m3 1024 维，RRF/rerank 天然融合。

> 为什么不给 meta 行零向量先入表、S 档再更新？——LanceDB 更新向量列笨重。改为：**L 档先把 meta 记录落 JSONL + 建独立 meta-bm25（0 嵌入即时可搜）；S 档再嵌入并把 meta 行 add 进表。** 这样"即时可用"和"表内统一"两不误（与现有 bm25 独立于表的架构一致）。

### 2.2 新增磁盘产物
```
data/
  meta/papers.jsonl          # L档：每行一篇的全字段记录（含无PDF篇）——仪表盘/分类/词法搜的即时数据源
  bm25_meta/                 # L档：meta文本的独立bm25（S档完成后可并入主bm25，或保留做纯词法兜底）
  stats_cache.json           # 仪表盘 /stats 的预聚合缓存（每档完成后刷新）
  categories/
    zotero_collections.json  # {bib_key: [collection路径]} + 收藏夹树+计数（读 zotero.sqlite）
    ai_topics.json           # 可选：AI 主题标签
  state/
    embedded_keys.txt        # 现有：F档已嵌入 stem
    meta_embedded.txt        # 新：S档已嵌入 meta 的 stem
    deep_index_queue.json    # 新：F档待深索队列 + 进度
  index_manifest.json        # 新：整库状态清单（各档进度、last_build_at、连接的.bib/zotero路径）
```

## 3. 建库流程重构（三个 Stage）

保留 `build_all.py` 作为"全量编排"，但拆出可独立调用的阶段脚本，并让 `server.py` 能按档触发。

### Stage L — `index_light.py`（即时词法，秒级）
输入：`.bib`。输出：`data/meta/papers.jsonl` + `data/bm25_meta/` + 刷新 `stats_cache.json`。
```
1. bibtexparser 解析全部条目（含无 PDF 的）。
2. 对每条：rec = meta_of(e) 扩展版（加 official_pages/has_pdf/itemtype/ingested_at；
   pages 从 e.get("pages") 取，has_pdf = bool(pdf_paths(e))）。
   text = clean(title) + "\n" + clean(abstract) + "\n" + de_emoji(keywords)
3. 写 papers.jsonl（一行一篇；key 用原始 bib key，另存 safe stem）。
4. jieba(载 legal dict) 分词 text → bm25s.index → 存 data/bm25_meta/ + bm25_meta_ids.json(存 key)。
5. 计算 stats_cache.json（见 §6 字段）。
```
- **0 模型加载、0 GPU/嵌入**；2000 条约几秒。这是"上来就能用"的核心。
- 复用 `extract.py` 的 `clean/authors/year_of/safe_name/build_legal_dict`（可把这些抽到 `textutil.py` 或 `bibutil.py` 共享，避免复制）。

### Stage S — `index_semantic.py`（快速语义，1–2 分钟，后台）
输入：`papers.jsonl` 中尚未在 `meta_embedded.txt` 的篇。输出：LanceDB 表里的 `row_type="meta"` 行 + 刷新 bm25。
```
1. 载 Embedder（bge-m3 ONNX）。
2. 批量 encode 每篇 text（max_length=512 足够摘要级）。
3. 组 meta 行（chunk_id="{key}::meta", page=None, vector=..., row_type="meta", 5新列),
   幂等 delete(key_predicate) 后 tbl.add；mark meta_embedded。
4. 重建主 bm25（覆盖 meta+chunk 全表 text），与现有 embed_index 的 bm25 段一致。
```
- 复用现有 `embedder.py`。表若不存在则 create（schema 以 meta 行为准，含全部列）。

### Stage F — 深度全文（数小时，opt-in/后台/按需）
= **现有 `extract.py → chunk.py → embed_index.py`**，几乎不改，只加：
- `extract.py`/`chunk.py`：把 4 个新列（itemtype/official_pages/has_pdf/ingested_at）带进块（从 meta 取）。
- `embed_index.py`：写块时 `row_type="chunk"`；其余不变（SAC 前缀逻辑保留）。
- **作用域参数**：支持只深索一个子集——
  - `--scope all`：全库有 PDF 的篇。
  - `--scope collection:<name>`：某 Zotero 收藏夹（从 categories 反查 key）。
  - `--scope key:<bibkey>`：单篇（按需触发用）。
  实现：在 extract 的 tasks 过滤处加 scope 过滤即可。
- **F 档完成一篇后**：在 `index_manifest.json` 标该 key `deep_indexed=true`；检索端据此把该篇结果从"仅摘要"升级为"全文可精读"。

### `build_all.py` 的新语义
```
python build_all.py --stage light      # 只 L
python build_all.py --stage semantic   # 只 S
python build_all.py --stage deep --scope all|collection:X|key:Y   # F
python build_all.py --stage all        # L→S→(不自动跑 F，除非 --deep)
默认（无参、每日增量）：L→S；F 只在用户显式要求或后台调度时跑。
```

### 后台调度与自愈（F 档）
- F 档由 `server.py` 起后台线程/子进程跑，写 `deep_index_queue.json` 进度；`GET /index/status` 读它。
- 长任务防休眠杀：注册**临时 Windows 计划任务**每 3 分钟检查 + 断点重起（模板见 memory `long-task-sleep-resilience`：`MultipleInstances=IgnoreNew` + 文件锁；别加 `-RunLevel Highest`，用 `-LogonType Interactive`）。跑完自删。
- 断点续：F 档天然断点续（extracted/chunks/embedded_keys 三级 done marker），杀了重跑即续。

## 4. 检索端改动（结果"自动变厚" + 徽章）

改 `retriever.py`（in-process 引擎）：

1. **就绪分级**：`load_all()` 时记录当前可用档位。
   - 若表不存在但 `bm25_meta` 存在 → **L-only 模式**：`search` 走 meta-bm25（纯词法），结果全部 `row_type=meta`。
   - 若表存在 → 正常 dense+bm25（表已含 meta 行、可能含 chunk 行）。
2. **统一查询 + 去重升级**：候选里同一 key 若**既有 chunk 行又有 meta 行**，优先保留 chunk 行（更具体），meta 行作兜底丢弃（避免同篇既出摘要又出正文）。仅有 meta 行的篇，返回 meta 行。
3. **结果增加字段**：`row_type`、`depth`（`"full"` if 该 key `deep_indexed` else `"abstract"`）。前端据此渲染徽章 `仅摘要` / `📄 全文可精读`。
4. **官方页码修复**：`citation(r)`：全文块用 `official_pages`（若有）而非 PDF 内 page；meta 行不带页码（或标"见全文"）。保留 PDF 页码到单独字段供跳转。
5. **无模式开关**：`sort` 参数照旧（relevance/tier/blend）；不新增浅/深参数。

前端 `app.js` 检索卡：加排序 `<select>`（相关性/期刊层级/混合）→ 传 `sort`；显示 author/year/官方页码 + tier badge（复用 `TIER_COLOR`）+ depth 徽章。

## 5. 首启向导（Onboarding）—— 产品的第一印象

新用户第一次打开 `http://127.0.0.1:8770`，若 `index_manifest.json` 不存在 → 进向导；否则直接进主界面。

### 步骤（全部非阻塞、可跳过）
```
① 欢迎 + 环境自检
   后端 GET /setup/detect →
     { bib: 自动探测到的 .bib 路径或 null,
       zotero_dir: 自动探测 Zotero 数据目录或 null,
       models_ready: bge-m3 是否就位, reranker_ready: bool }
   探测顺序：环境变量 LOCALKB_BIB → %USERPROFILE%\Zotero\*.bib → 让用户"浏览选择"。

② 连接文库
   POST /setup/connect { bib_path, zotero_dir? } → 校验可读、条目数；写 index_manifest。

③ 即时索引（Stage L，秒级，带进度）
   POST /index/light → 跑 index_light.py → 返回 { papers, with_pdf, no_pdf }
   完成即跳主界面：仪表盘点亮 + 检索框可用（此刻是词法）。

④ 后台提质（Stage S，自动，不挡路）
   连接成功后自动 POST /index/semantic（后台）；顶部细进度条
   "正在提升检索质量… 1240/2119"，读 GET /index/status。

⑤ 邀请深度索引（Stage F，用户决定）
   一张卡片：「深度索引让回答精确到页码。现在开始？」
     [深索全库（约 N 小时，可后台）] [只选收藏夹…] [以后再说]
   选择 → POST /index/deep { scope }。

⑥ 可选设置（随时可回来）
   - LLM key（chat 用；provider/base/model/key，存 localStorage + 本地）
   - MinerU token（高精度提取；去 mineru.net/apiManage/token 生成，存本地）
```

### 首启后主界面状态提示
- 顶栏状态 pill 分档显示：`词法就绪` → `语义就绪·N篇` → `全文 X/N 篇`。
- 深索进行中：顶部常驻细进度条，可点开看 `GET /index/status`。

### 新增后端端点（都加在 `server.py`）
| 端点 | 作用 | 返回 |
|---|---|---|
| `GET /setup/detect` | 自检 .bib/zotero/模型 | 见上 |
| `POST /setup/connect` | 保存连接、校验 | `{ok, entries}` |
| `POST /index/light` | 跑 Stage L | `{papers, with_pdf, no_pdf}` |
| `POST /index/semantic` | 起 Stage S（后台） | `{ok}` |
| `POST /index/deep` | 起 Stage F（后台，带 scope） | `{ok, queued}` |
| `GET /index/status` | 三档统一进度 | `{stage, papers, meta_done, deep_done, deep_total, running, log[-50:]}` |
| `POST /paper/{key}/deep` | 单篇按需深索（Beaver 式） | `{ok}` |
| `GET /stats` | 仪表盘聚合（读 stats_cache） | 见 §6 |
| `GET /categories` | 收藏夹树 + AI 主题 | 见 §7 |

> `GET /index/status` 统一替代脆弱的 `/build/status`（内存日志）。进度来自磁盘（papers.jsonl 计数 / meta_embedded.txt / embedded_keys.txt / deep_index_queue.json），重启不丢、CLI 触发也可见。

### 按需深索（回答"等他们决定了再弄"的第三种触发）
用户在某篇上点"深度问答"而该篇 `deep_indexed=false` 时：前端先 `POST /paper/{key}/deep`（只深索这一篇，秒级~十几秒），完成后再问答。这就是 Beaver 式"用到哪篇才深索哪篇"，零全库等待。

## 7. 分类：Zotero 收藏夹（读 sqlite）为主 + AI 主题为辅

**从 `.bib` 读收藏夹不可能**（Better BibTeX 默认不导出，全库 0 个 collection 字段）。改从 **`zotero.sqlite`** 读（实测 join 成功率 98%、覆盖约 1413 篇）。

### `build_categories.py`
```
1. 定位 zotero.sqlite（默认 zotero_dir\zotero.sqlite；产品要自动探测+让用户指定）。
2. ⚠️ Zotero 开着时 sqlite 被独占锁 → 先复制成只读副本到 temp 再打开（sqlite3 只读连接）。
3. join：collections → collectionItems → items → itemAttachments(附件 item 的 8位 key)
   附件 key = storage 目录名 = .bib 里 file 路径中的 storageKey → 反查 bib key。
4. 输出 data/categories/zotero_collections.json：
   { "tree":[{name,path,parent,count,children:[...]}], "by_key":{bib_key:[path,...]} }
```
- **存 sidecar，不进表、不加向量列、不重嵌。** 检索结果已带 `key`，贴分类只需查表。
- 退路（拿不到 sqlite / 换机器）：引导用户在 Better BibTeX 开 `keywords = Tags+Collections` 导出，则收藏夹进 keywords 字段。

### AI 主题（可选，为辅）
- 复用 LanceDB 现成 bge-m3 向量：按 key 取均值 → k-means/HDBSCAN → 每簇喂代表标题给 LLM 起名（约 30 次调用）。只用来**补 sqlite 未覆盖的散篇 + 跨收藏夹鸟瞰**。存 `ai_topics.json`。
- 增量时新篇就近分配到已有簇质心，无需重跑 LLM。

### 呈现
- 新「浏览」tab：左侧收藏夹层级树（带计数）+ AI 主题平铺 chips（用中性紫，区别于 tier 配色）；选中以可关闭 chips 呈现。
- `POST /search` 加可选 `collections`/`topic` 过滤参数：**分类=WHERE（按 key 筛候选池），tier=ORDER BY**，两者正交，不改 `journal_tiers.py`。

## 8. MinerU 可选高精度提取后端

- **定位**：只在 F 档（全文提取）起作用，专治中文双栏/复杂版式 PDF。默认仍用本地 PyMuPDF（免 token、离线）。
- **用户自备 token**：设置里贴 mineru.net/apiManage/token 生成的 token，存本地（同 LLM key 模式：直连 MinerU、不经开发者服务器，隐私一致）。
- **接法**：`extract.py` 加一个 `EXTRACTOR` 分支——若 MinerU 开启且（该 PDF 用 PyMuPDF 提取质量差 / 用户对该篇强制）→ 调 MinerU API 拿高保真 markdown，其余照旧。速率/费用是用户自己的。
- 列为**备选项**，非默认路径。

## 9. 产品化工程（分发给用户必须解决）

1. **打包**：内嵌 Python 整合包 + pywebview（已调研可行，砍 torch 后约 1.7GB）。
   - L/S 档只需 `bge-m3` ONNX；reranker 仅 F 档需要 → 可延后下载。
   - 选项：安装包小 = 首启后台下载模型；或整包带模型。
2. **模型/路径参数化**：`config.MODELS`/`BIB` 已支持环境变量；再加 UI 里可配。**别 mkdir MODELS**（只读）。所有写入只落用户数据目录。
3. **多用户数据目录**：`data/` 迁到 `%APPDATA%\LocalKB` 或用户可选，避免装在 Program Files 下写权限问题。
4. **zotero.sqlite 锁 + 路径探测**（见 §7）：只读副本 + 自动探测 + 手动指定兜底。
5. **首启质量依赖题录**：约 26% 篇无摘要，L 档只有标题+关键词——空状态诚实提示"该篇仅题录，深索后可全文检索"。
6. **隐私**：LLM key / MinerU token 只存本机；默认全离线（除用户主动用云 LLM/MinerU）。
7. **编码坑**：中文路径 `.bat` 存 GBK；build 子进程编码兜底（见 §1.4）。

## 10. 分阶段实现路线图（每阶段可独立交付+验收）

> 排序原则：先让产品"上来就能用 + 能看见"，再逐层加深。

### Phase 1 — 即时可用 + 可视化（核心 MVP）
**目标**：新用户连上 → 秒级能搜 + 仪表盘点亮。
- 写 `index_light.py`（Stage L）；抽公共 bib 工具到 `bibutil.py`。
- `retriever.py` 加 L-only 词法模式（表不存在时走 meta-bm25）。
- `server.py` 加 `/setup/detect` `/setup/connect` `/index/light` `/index/status` `/stats`。
- 前端：首启向导步骤①②③ + 「📊库总览」tab（手绘 SVG，读 `/stats`）。
- **验收**：全新目录启动 → 向导连 .bib → 几秒内检索框能返回结果、仪表盘显示全库分布（含无 PDF 篇）。

### Phase 2 — 语义提质（S 档 + 统一表）
- 新 schema 建表（含 `row_type` 等 5 列）；写 `index_semantic.py`。
- `retriever.py` 统一查询 + 同 key chunk 优先 + depth 徽章字段。
- 前端：顶部"正在提升检索质量"进度条；结果卡 `仅摘要` 徽章。
- **验收**：连上约 1–2 分钟后，语义查询（近义词）能召回；结果标注档位。

### Phase 3 — 全文深索（F 档 opt-in/后台/按需）
- `extract/chunk/embed_index` 带 4 新列 + `row_type="chunk"` + `--scope`。
- `server.py` 加 `/index/deep` `/paper/{key}/deep`；后台调度 + 自愈计划任务。
- 前端：向导步骤⑤ + 单篇"深度问答"触发按需深索 + `📄全文可精读` 徽章。
- **验收**：点"深索某收藏夹"→ 后台跑、进度可见、完成后该批结果升级为页码级；合盖/重启能断点续。

### Phase 4 — 分类 + 期刊排序 + 官方页码
- `build_categories.py`（zotero.sqlite 只读副本 + 98% join）；`/categories` + 浏览 tab + `/search` 分类过滤。
- 检索卡接 tier 排序 `<select>`；显示 author/year。
- `meta_of` 扩展保留 `official_pages`；`citation()` 用官方页码。
- **验收**：按收藏夹筛选、按期刊层级排序、引用显示官方页码。

### Phase 5 — MinerU + 打包分发
- MinerU 可选后端 + token 设置。
- pywebview 打包、模型分发策略、数据目录迁移、多用户路径。
- **验收**：在一台干净机器上安装运行，走完首启到检索全流程。
