# PaperPiggy 全类型文献分级与工作流改造任务书

> 状态：待新对话实施。本文是用户已拍板的实施任务，不是讨论稿，也不是 `_archive/` 中的历史方案。
>
> 决策日期：2026-07-18

## 一、实施目标

把当前以“期刊档位”为中心的功能，改造成覆盖期刊论文、书籍、书章、学位论文、法源、案例、标准、报告、数据集、预印本、会议论文、网页等全部来源的统一评价体系，并同步改造浏览页、库总览、综述页和两个 Agent 工作流。

最终必须完成开发、迁移兼容、测试、用户指引、版本更新、安装器构建、Git 推送和 GitHub Release 发布，不以“代码已改”为终点。

## 二、不可再重新讨论的产品决定

### 2.1 普通页面只显示一个客观标签

检索、浏览、综述来源等普通页面不再以“权威/准权威/核心”等主观档位作为主徽标，只显示一个最高优先级的客观标签。

示例：

- `三大刊 · CLSCI · CSSCI` → 只显示 `三大刊`
- `CLSCI · CSSCI` → 只显示 `CLSCI`
- `SSCI · JCR Q1` → 只显示 `SSCI Q1`
- `CSSCI来源 · 北大核心` → 只显示 `CSSCI`
- `CSSCI扩展 · 北大核心` → 只显示 `CSSCI扩展`
- 顶尖外文法评同时命中 SSCI/JCR → 只显示 `顶尖法评`

“法学三刊”一律更名为 **“三大刊”**。

### 2.2 法学（开发者增强）只保留四个评价档位

四档固定为：

1. 权威
2. 顶级（原“准权威”）
3. 核心
4. 普通

不再显示“次核心”“一般”“待确认”“警示”等第五档或第六档。待确认、一般、未知均归入“普通”，但内部可以保留更细的排序分数和识别解释。

旧档无损折叠：

- T1 → 权威
- T1b → 顶级
- T2/T3 → 核心
- T4/T5/待确认/未知 → 普通

CSSCI扩展必须高于北大核心。两者可以都属于“核心”，但内部排序不得相同；具体分差保持可配置，后续再结合真实金标查询校准。

### 2.2a 增加娱乐命名学科“法学（开发者增强：夯到拉）”

新增一个可选学科：**法学（开发者增强：夯到拉）**。

它与“法学（开发者增强）”在目录、映射、规则、权重、内部档位、排序、检索加成、手动改档、工作流和后续更新方面必须完全相同，唯一差别是四档的显示名称：

| 内部标准档位 | 法学（开发者增强） | 法学（开发者增强：夯到拉） |
|---|---|---|
| 权威 | 权威 | 夯 |
| 顶级 | 顶级 | 顶级 |
| 核心 | 核心 | 人上人 |
| 普通 | 普通 | NPC |

实现上不得复制出第二套独立目录、映射或权重配置。娱乐学科必须继承或引用“法学（开发者增强）”的同一份规则，只在最终展示层替换档位名称；以后更新基础学科时应自动同步生效，不得要求维护者手工改两份。

用户选择该学科时，在设置页和库总览显示清楚、但不过度打扰的说明：

> 仅供娱乐：本学科与“法学（开发者增强）”的评级规则、权重、排序和功能完全相同，只替换四档显示名称；娱乐称呼不代表对论文、作者或研究价值的评价。

普通页面的客观标签（如“三大刊”“CLSCI”“SSCI Q1”“书籍”“学位论文”）不受娱乐命名影响。需要显示主观档位的库总览、悬停解释、手动改档菜单、工作流文献表等位置，按当前所选学科显示“夯/顶级/人上人/NPC”。数据存储和接口内部应继续使用稳定的标准档位代码，切换两个学科不能产生重复缓存、重复改档记录或不可逆迁移。

### 2.3 书籍和学位论文

- `book`：全部进入 **权威**，显示 `书籍`。
- `bookSection`：视为书籍的一部分，同样进入 **权威**，显示 `书章`。
- `thesis`：全部进入 **顶级**；有学位类型时显示 `博士论文`/`硕士论文`，信息不足时显示 `学位论文`。

这套规则是法学（开发者增强）的用户预设，不得以“通用学术评价通常不会这样分”为由擅自改回保守档位。用户明确说明：其 Zotero 中保存的书籍均为重要材料。

### 2.4 台湾期刊

当前 `tw_law.json` 是个人旧目录，不是正式 TSSCI 全表。保留现有特殊偏好：

- 重点三刊：中正大学法学集刊、台湾大学法学论丛、政大法学评论 → **权威**。
- 当前个人目录中的其他10刊 → **核心**。
- 后续引入正式 TSSCI 法学期刊表：不在重点三刊中的 TSSCI 法学期刊 → **顶级**。

客观标签规则：

- 只有确实命中正式 TSSCI 数据时才显示 `TSSCI`。
- 个人目录命中但未命中正式 TSSCI 时显示 `台湾法学`，不能冒充官方收录。
- 重点三刊如同时命中 TSSCI，普通页面仍显示客观标签 `TSSCI`，其“权威”档位在库总览、悬停解释和改档弹窗中体现。

### 2.5 外文法学来源

保留法学（开发者增强）现有“外刊不打折”和高权重偏好，不得因四档合并而降低：

- 顶尖外文法评 → 权威。
- 当前 `ssci_law_authority` 精选外文权威期刊 → 顶级。
- 其余 SSCI/JCR/SJR 来源按法学增强映射进入四档。

ShowJCR提供 SSCI/JCR 分区等客观标签，不能覆盖“顶尖法评”“精选外文权威”的更高学科判断。

### 2.6 其他非期刊来源预设

- 法律法规、司法解释、正式规范性文件 → 顶级，标签 `法源`。
- 案例 → 沿用当前预设进入顶级，标签 `案例`；若能可靠识别，可细分 `指导案例`、`典型案例`、`裁判文书`，但不得让不可靠推断阻塞本轮。
- 标准 → 顶级，标签 `标准`。
- 报告、白皮书 → 核心；能确认官方性质时显示 `官方报告`，否则显示 `研究报告`。
- 权威机构数据集 → 核心，标签 `权威数据`；其他数据集 → 普通，标签 `数据集`。
- 预印本 → 普通，标签 `预印本`。
- 会议论文 → 普通，标签 `会议论文`。
- 普通网页、博客、报纸、文件 → 普通；若内容实际是法源、报告或数据集，按内容性质覆盖容器类型。

手动改档优先级始终最高，但只改变四档评价，不伪造或覆盖客观标签。

## 三、统一判定模型

不要继续让所有材料都经过“期刊名→期刊目录”的单一路径。统一流程应为：

1. 识别真实文献性质。
2. 收集所有客观目录信号。
3. 按当前学科映射为四档评价和内部权重。
4. 从候选客观标签中只选一个最高标签用于普通页面。
5. 应用单篇手动改档；手动改档只改评价层。

必须提供一个单一事实源，统一返回至少以下信息：

- 文献性质；
- 唯一客观标签；
- 四档评价；
- 内部权重与排序；
- 命中目录/规则解释；
- 是否手动改档。

浏览卡、检索结果、库总览、综述来源、MCP/Agent输出均调用这一事实源，不得各自复制一套标签优先级。

## 四、客观标签选择规则

标签选择遵循“先比较学科映射后的档位，再按同档目录优先级决胜”。至少覆盖：

`三大刊`、`顶尖法评`、`TSSCI`、`CLSCI`、`SSCI Q1/Q2/Q3/Q4`、`CSSCI`、`CSSCI扩展`、`北大核心`、`SJR Q1/Q2/Q3/Q4`、`台湾法学`、`书籍`、`书章`、`博士论文`、`硕士论文`、`学位论文`、`法源`、`案例`、`标准`、`官方报告`、`研究报告`、`权威数据`、`数据集`、`预印本`、`会议论文`、`网页`、`报纸`。

SSCI与JCR分区组合显示为一个标签，如 `SSCI Q1`，不能同时堆两个徽标。

## 五、数据与目录来源

### 5.1 期刊目录

- 中文目录基于 `cnki-Scholar`。
- 外文 SSCI/JCR 数据基于 `ShowJCR`。
- 保留项目内的三大刊、CLSCI、顶尖外文法评、精选外文权威和台湾个人目录。
- 引入正式 TSSCI 法学期刊数据，必须记录来源、版本、检查日期，不得把个人目录伪装成 TSSCI。

每个目录记录：来源URL、上游版本或提交、最后检查日期、下次检查日期。每半年检查一次上游变化，生成差异；更新目录不得覆盖用户手动改档。

利用项目现有定时任务机制建立半年检查任务，避免另造互不相通的调度系统。

### 5.2 Zotero元数据补全

当前索引没有保留评价非期刊材料所需的若干字段。建库时应尽量带入并贯通：

- URL、网站名称、访问日期；
- 出版社、出版地、ISBN、版次、丛书；
- 母书名；
- 学位授予单位、学位类型；
- 报告机构、报告类型、报告编号；
- 会议名称、论文集；
- 法院、案号、裁判日期；
- 标准号、版本等。

迁移必须兼容现有 `papers.jsonl`：旧记录缺字段时正常降级，不得要求用户删除库或全量重建后才能启动。若需要重建才能补齐新字段，应在设置页给出明确提示，并验证“清空并从头重建索引”保留用户数据。

## 六、浏览页改造

### 6.1 文献类型筛选

在现有收藏/分类下拉框旁增加独立的“全部文献类型”筛选，至少包含：

- 期刊论文
- 书籍
- 书章
- 学位论文
- 法源
- 案例
- 标准
- 报告与白皮书
- 预印本
- 会议论文
- 数据集
- 网页与其他

显示过滤后的数量，支持与收藏、主题、深索状态、排序叠加，支持分页。

筛选按“识别后的真实文献性质”而不是简单照抄 Zotero itemType。例如网页中的白皮书应进入“报告与白皮书”，报纸中刊载的正式司法意见应进入“法源”。

后端 `/papers` 增加稳定的类型过滤参数；MCP列举文献的能力也应提供等价过滤，避免只在前端实现。

### 6.2 卡片与手动改档

- 卡片只显示一个客观标签。
- 点击标签或现有改档入口，弹出 `权威/顶级/核心/普通/恢复自动`。
- 改档后客观标签不变，增加轻量的手动标记；悬停或弹窗中说明“已手动设为××”。
- 普通、未知、题录不完整均不显示“待确认”或机器人警示档位。

## 七、库总览改造

库总览是四档评价的主要展示位置，至少包括：

- 权威、顶级、核心、普通的数量与比例；
- 客观目录/文献性质到四档的映射；
- 内部权重；
- 用户可调整的映射；
- 用户单篇手动改档；
- 目录来源与更新时间；
- 半年检查状态；
- 按客观标签、文献性质查看明细。

用户修改某个目录映射时，只改变当前学科下的评价，不篡改客观目录数据。

## 八、综述页改造

- 将大块黄色 `🤖 未核验` 从正文上方移走，缩成标题右侧状态区的小标签，与“标为已核验”按钮放在一起。
- 综述的每条“参考来源”前显示统一的唯一客观标签和四档级别；四档可用低干扰文字或悬停展示，不能堆叠多个目录徽标。
- 参考来源顶部显示来源构成摘要，例如 `三大刊 1 · CLSCI 12 · 书籍 2`。
- 综述页接口应按 source key 查回统一评价结果，不能只传 citation 字符串。
- 列表页中的未核验状态移至卡片右侧状态区，不插在元数据中间。

## 九、两个 Agent 工作流

同步修改：

- “写论文与综述”；
- “跨学科发散与补文献”。

共同要求：

1. 第一优先级是从库内已有文献的参考文献反向追踪。
2. 候选必须记录来自哪篇库内文献、哪条参考文献；可行时保留页码或参考文献序号。
3. 扩展现有仅偏中文的参考文献挖掘，使其能识别外文题名、作者、年份、DOI和期刊。
4. 先完成库内参考链追踪、去重和缺口判断，再联网搜索。
5. 联网候选与库内参考链候选分区展示，不能混为一谈。
6. 同等相关时优先高等级外文和台湾文献；每个重要论题都要检查外文理论、比较法和实证研究。
7. 如果最终没有使用外文或台湾材料，必须说明检索过什么以及为何不适用，不能机械凑数量。
8. 不得因“出现在真实参考文献中”就省略核验；OCR可能破坏题名、年份和作者，仍需核对原文或 DOI/正式目录。

两个工作流的最终交付物必须新增“文献使用与缺口”一节，列出：

- 已使用文献；
- 唯一客观标签；
- 四档级别；
- 语言/地区；
- 用于哪个部分；
- 仍需补充的文献；
- 候选来源链；
- 是否已入库；
- 是否已核验；
- 联网补充项及其来源链接。

修改任何 `agent_ws._WF_*` 模板后，必须按 AGENTS.md 要求运行 `agent_ws.py --print-hashes`，把新 hash 追加到 `_FACTORY_HASHES`，旧 hash 一个都不能删除。

## 十二、发布要求

用户已明确授权本任务完成后发布，不要在新对话中再次询问“是否发布”。但若 GitHub 认证失效、构建失败或存在真实阻断，必须诚实报告。

3. 以执行时 `config.APP_VERSION` 为事实源递增补丁版本；当前任务书生成时版本为 1.0.23，但执行时不得盲信此数字。

7. 核对发布产物、版本和更新清单。
8. 提交、push 到 GitHub。
9. 创建对应版本的 GitHub Release；上传明确列出的本版产物，禁止使用会带入旧文件的通配符。
10. 发布后再次核对 Release 页面、安装器和应用内更新所需文件是否齐全。

最终报告必须列出：版本号、commit、分支、测试结果、安装器路径、Release链接、未完成或降级项。只有 Release 真正发布成功才能声称任务完成。
