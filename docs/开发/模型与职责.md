# 模型与职责 —— 谁负责哪一部分

LocalKB 里有**三类模型**，分工清晰、互不替代：

| 模型 | 类型 | 负责什么 | 何时用 |
|---|---|---|---|
| **bge-m3** | 嵌入(embedding) | 把文本变成 1024 维向量，做**语义检索**（按意思找，不只按关键词） | 建索引时嵌入每个正文块/题录；查询时嵌入用户问题 |
| **bge-reranker-v2-m3** | 重排(rerank) | 对初筛出的候选段落**精排**（比初筛更准地判断相关性） | 仅查询时：top-50 候选 → 精排 → top-8 |
| **你的 LLM key**（DeepSeek/SiliconFlow…） | 生成(LLM) | **写答案 / agent 推理**：读检索到的片段，生成带引用的自然语言回答 | 仅"对话"功能 & agent 访问时；**不参与检索、不参与建索引** |

## 一句话分工
- **bge-m3 + reranker = "找得准"**（检索引擎，本地或 API）
- **LLM key = "说得好"**（把找到的材料组织成回答；换更强的 LLM 只影响回答质量，不影响检索）

## 深索（全文深度索引）的建构过程用哪些模型？
深索 = 把一篇 PDF 变成可页级检索的向量。流程：

```
PDF ──PyMuPDF提取文字(无模型)──▶ 规则切块(无模型) ──▶ bge-m3 嵌入每块 ──▶ 存入向量库
                                                    ▲
                                        （唯一吃算力/耗时的一步）
```

- **提取**：PyMuPDF / pymupdf4llm（纯解析，不用模型）
- **切块**：结构感知规则（不用模型）
- **嵌入**：**只用 bge-m3**（这是耗时大头——全库 19 万块原来跑了约 5 小时）
- **重排模型不参与建索引**（它只在查询时用）
- **LLM 不参与建索引**（例外：可选的 SAC 文档摘要前缀，本项目里深索已带，非必需）

> 结论：**深索建构只依赖 PyMuPDF + bge-m3**。reranker 和 LLM 都是查询期的事。

## SiliconFlow 的免费模型能胜任吗？
**能，全部能**：
- `BAAI/bge-m3`（嵌入）——**免费**，同款模型、1024 维一致 → 建索引 + 查询都能用。
- `BAAI/bge-reranker-v2-m3`（重排）——**免费** → 精排能用。
- 对话/agent 的 LLM——SiliconFlow 也有**免费**对话模型（如 DeepSeek-V3）。

也就是说，一个 SiliconFlow key 可以同时供 **嵌入 + 重排 + 对话** 三用，全免费。

**代价 / 注意**：
- **一致性铁律**：建索引和查询必须用**同一后端**。本地 bge-m3(INT8) 与 API bge-m3(全精度) 的向量略有差异，混用会掉点。要么全本地、要么全 API（切换需重建语义/深索索引）。详见 [数据格式与深索迁移.md](数据格式与深索迁移.md)。
- **深索走 API 的量**：全库 19 万块若全走 API 嵌入，免费档限流下会较慢。但 LocalKB 是**选择性深索**（只做你勾选的少量文献），调用量可控，实际很顺。
- **隐私**：API 模式建索引时会把 PDF 正文发给 SiliconFlow；看重"不出本机"就用本地模式。

## 给未来开发的提醒
- 换 LLM（对话）随便换，不影响已建索引。
- **不要动 bge-m3 的嵌入管线**（模型/池化/归一/维度），否则旧向量全部作废、要重跑几小时。这是深索兼容的红线，见迁移文档。
