## 1. 目标与范围

### 要解决什么
F38 根因：用户看到的**主徽标**（`tierBadge`，app.js:141-143）吃的是**建库时烤死的扁平字段** `journal_tier`（`journal_tiers.py` 出品，无学科概念），所以设置里锁定的学科（`law` / `law_personal`）对主徽标完全不可见——"台湾核心""外文顶级法评"这些个人决策全局生效、且与当前学科脱钩。

产品拍板方案 B「统一治本」：**废弃旧扁平表作为徽标真相源，让所有面向用户的期刊档位显示都变成学科感知**，口径统一到 `journal_grading` 引擎（`resolve_journal_weight`）。三处口径要一并收敛：
- 检索结果卡主徽标（app.js:141-143/183）
- 浏览页文献卡徽标（app.js:644）+ 推荐分 `_rec_score`（server.py:338-350）
- 库总览「期刊分级分布」`by_tier` + 「高频期刊」`by_journal`（app.js:300-331，数据 index_light.py:65/77-79）

并顺带做两项**正确性加固**：
- 落实 `disciplines[*].catalogs` 白名单，把当前"巧合式正确"（靠原型 map 恰好不引用私有目录）变成"强制正确"，堵住 `explain.signalSet/hitCatalogs` 的台湾信号泄漏（resolver.py:106-107,175）。
- 明确旧扁平表 `journal_tiers.py/json` 的过渡去留。

### 不含什么
- 不改 `journal_grading` 的档位算法/系数/原型 map（config 已就位，见下文核实结论）。
- 不改检索召回/RRF/rerank/排序策略本身（`_apply_sort`、`_blend_bonus` 已在吃 `journal_weight`，retriever.py:364-382，无需动）。
- 不新增学科；不做多学科并存（仍是"整库锁定单学科"）。

### 2.3 关键结论②：浏览页 + 库总览走存储 `journal_tier`，不随学科变 —— ✅ 成立（这是真正的难点）

## 4. 后端接口

### 4.0 方案选型（难点：A 现算 vs B 重建入列）—— 推荐 A

| | A：查询/加载期按当前学科现算 | B：重建索引把 weight_tier/journal_weight 存进列 |
|---|---|---|
| 与"改学科即时生效"设计一致性 | ✅ 完全一致（settings 实时读，retriever.py:131-136） | ❌ 冲突：切学科就得全库重建，倒退成"烤死" |
| 迁移成本 | 无需重建，前端+少量后端聚合改动 | 全库重建 + schema 迁列 + 双写过渡 |
| 运行开销 | 分布/浏览按**去重刊名**算，可 memo（见下） | 建库期一次性 |
| 风险 | 需评估全库现算耗时 | schema 变更、旧索引兼容 |

**推荐 A**，决定性理由：既有架构的核心卖点是"改设置即时生效、不用重建"（settings.py:86-88、server.py:243-249 注释明说）。B 把 `weight_tier` 烤进列会让"切学科"重新变成"要重建"，与产品重构方向（学科可切）直接矛盾。A 的唯一顾虑是性能，而性能可控——**分布只需按去重刊名算，不是按 total papers**：

- `resolve_journal_weight` 对 ISSN/精确刊名命中是 O(1)（identify.py:62-77 走 dict）；只有 miss 才触发 fuzzy（identify.py:79-88 遍历 `by_name`，O(目录规模)）。
- 库里去重刊名通常几百~数千（`compute_stats` 的 `jc`/`jt` 已按刊名去重，index_light.py:68-69）。对**去重刊名**各算一次并 memo，即可覆盖全库分布与浏览卡。

> **需核实（性能基线）**：① `by_name` 索引条数——跑 `python journal_grading/loader.py` 看输出（loader.py:208-212 会打印 ISSN/归一名条数）；② 全库去重刊名数——看 `stats_cache.json` 的 `by_journal` 或 `wc -l meta/papers.jsonl` 估算。若去重刊名 < ~5000 且 miss 率不高，A 的一次性冷算在百毫秒~一两秒量级，且可缓存。B 仅在"分布计算被证明是热点且无法缓存"时才回退考虑。

### 4.1 新增共享服务模块 `grading_svc.py`（后端统一入口）

新建 `D:\...\LocalKB源码\grading_svc.py`，把"按当前学科算某刊 weight/tier + 中文档名 + 带 memo"收敛到一处，供 `server.py` 复用（retriever.py 已自有 `_weight_res`，不动）：

```python
# grading_svc.py —— 期刊分级的服务层门面（学科感知 + memo + 中文档名映射）
import settings as S
try:
    import journal_grading as JG
except Exception:
    JG = None

TIER_CN = {"T1":"权威","T1b":"准权威","T2":"核心","T3":"次核心",
           "T4":"一般","T5":"普通","待确认":"待确认"}
TIER_RANK = {"T1":0,"T1b":1,"T2":2,"T3":3,"T4":4,"T5":5,"待确认":6}

_MEMO = {}          # (disc, journal, issn) -> {"tier","weight","cn","rank","needs_review"}
_MEMO_DISC = None

def _disc():
    return S.discipline()

def grade(journal, issn=""):
    """按当前锁定学科算一刊分级。引擎缺失→回退 None（调用方兜旧 journal_tier）。"""
    if JG is None:
        return None
    disc = _disc()
    global _MEMO_DISC
    if disc != _MEMO_DISC:            # 学科切换即清缓存（settings 变了）
        _MEMO.clear(); _MEMO_DISC = disc
    k = (disc, journal or "", issn or "")
    if k in _MEMO:
        return _MEMO[k]
    try:
        r = JG.resolve_journal_weight({"journal": journal or "", "issn": issn or ""}, disc)
        t = r.get("tier") or "待确认"
        out = {"tier": t, "weight": r.get("weight"),
               "cn": TIER_CN.get(t, t), "rank": TIER_RANK.get(t, 6),
               "needs_review": bool(r.get("needsReview"))}
    except Exception:
        out = None
    _MEMO[k] = out
    return out
```

### 4.2 `GET /stats` —— by_tier / by_journal 改按学科现算
落点：`server.py:305-314 stats_ep()`。保留 `stats_cache.json` 的其余字段（coverage/by_year/by_lang…），**只重算 by_tier / by_journal**，用 `_load_papers()`（server.py:318-328，已全量缓存 papers.jsonl）现聚合：

```python
import grading_svc as GS
from collections import Counter

def _weight_dist(papers):
    """按当前学科重算期刊分级分布（按去重刊名 memo）。返回 (by_tier, by_journal)。"""
    tier_n = Counter(); jn = Counter(); jtier = {}
    for p in papers.values():
        j = p.get("journal", "")
        g = GS.grade(j, p.get("issn", "")) if j else None
        cn  = g["cn"]   if g else (p.get("journal_tier") or "未知")   # 引擎down兜旧
        rnk = g["rank"] if g else 6
        tier_n[(cn, rnk)] += 1
        if j:
            jn[j] += 1; jtier[j] = (cn, rnk)
    by_tier = sorted(({"tier": cn, "rank": r, "n": n} for (cn, r), n in tier_n.items()),
                     key=lambda x: x["rank"])
    by_journal = [{"journal": j, "tier": jtier[j][0], "rank": jtier[j][1], "n": n}
                  for j, n in jn.most_common(15)]
    return by_tier, by_journal

@app.get("/stats")
def stats_ep():
    if not C.STATS_CACHE.exists():
        return JSONResponse({"error": "尚未建立索引"}, status_code=404)
    s = json.loads(C.STATS_CACHE.read_text(encoding="utf-8"))
    ek = C.STATE / "embedded_keys.txt"
    if ek.exists() and isinstance(s.get("coverage"), dict):
        s["coverage"]["deep_indexed"] = len(ek.read_text(encoding="utf-8").split())
    # F38-B：期刊分级分布按当前锁定学科现算（不随建库烤死）
    try:
        pap = _load_papers()
        s["by_tier"], s["by_journal"] = _weight_dist(pap)
        s["grading_discipline"] = S.discipline()   # 前端可显示"当前学科：法学"
    except Exception as e:
        pass   # 现算失败则退回 stats_cache 里的旧分布，不 500
    return s
```

`_rec_score` 改吃连续权重（0–1）替代离散 `tier_rank`，量纲对齐旧公式（旧 `(6-tr)*2.0` 峰值 12，新用 `weight*10` 峰值 10，近似）：
```python
def _rec_score(p, g=None):
    """值得读打分：期刊权重为主 + 新近度 + 有 PDF。学科感知(g)优先，缺则回退旧 tier_rank。"""
    if g and g.get("weight") is not None:
        s = g["weight"] * 10.0            # 0–1 → 0–10，替代旧 (6-tier_rank)*2
    else:
        s = (6 - p.get("tier_rank", 6)) * 2.0   # 引擎down兜旧
    try:
        yr = int(p.get("year") or 0)
    except Exception:
        yr = 0
    if yr >= 2015:
        s += min(3.0, (yr - 2015) * 0.3)
    if p.get("has_pdf"):
        s += 1.0
    return round(s, 2)
```

> 注意：`papers()` 对**全部 papers** 算 `_rec_score` 再排序取 `limit`（server.py:465-485）。开 memo 后新增成本 = O(去重刊名)，与 §4.2 同规模，可接受。

## 9. 风险与未决点

1. **性能（方案 A 的核心假设，需核实）**：全库分布/浏览现算依赖"去重刊名数不大 + ISSN/精确命中率高（少走 fuzzy）"。**核实方法**：`python journal_grading/loader.py` 看 `by_name` 条数；`wc -l meta/papers.jsonl`；给 `stats_ep` 加临时计时日志跑真实库。**缓解**：`grading_svc` 已按刊名 memo；若仍慢，给 `_weight_dist` 结果再加一层"按 (discipline, PAPERS_JSONL.mtime) 缓存"（papers 不变 + 学科不变则复用），只在切学科或重建后失效。此项若不达标才考虑方案 B。

2. **检索卡 vs 浏览卡字段口径**：检索侧 `weight_tier` 现为引擎原始 `T?` 码（retriever.py:152），浏览侧新给中文。T8 必须统一（否则同一 `tierBadge` 两种数据）。**已在 §5.1/T8 指定统一为中文**，实施时勿遗漏检索侧映射。

3. **light（词法）模式**：`search_light` 已调 `_weight_res`（retriever.py:332,349），主徽标同样能吃 `weight_tier`。但其排序回退仍读旧 `journal_tier`（retriever.py:126-128）——本方案不改排序，仅改显示，**light 模式徽标可正确、排序仍旧口径**，属可接受的已知不一致（排序不在 F38-B 范围）。若要彻底统一，需另立任务改 `_tier_of`/`_apply_sort` 的 light 分支。

4. **fallback 语义边界（易错）**：白名单裁剪后"识别到但可见信号空"必须落「普通/其他核心」而非「待确认」（§4.4）。写错会让标准 law 下所有台湾/外文法评刊变「待确认」灰徽标（体验倒退且 selftest:175 会红）。实现时以 T4 断言把关。

5. **`issn` 字段可得性**：`grading_svc.grade` 传了 `p.get("issn","")` 提升识别精度，但需确认 `papers.jsonl`/检索记录里确有 `issn` 字段。**需核实**：grep `papers.jsonl` 首行是否含 `issn`。若普遍缺失，identify 退化为按刊名匹配（identify.py:68-88），功能不受损但 fuzzy 命中率略降、性能略差——不阻塞上线。

6. **`by_tier` 桶语义变更对用户认知**：库总览分布桶名从「CLSCI/CSSCI/普刊」变成「权威/核心/普通」，且**同一库在不同学科下分布不同**。这是产品意图（学科感知），但需在副标题明示"按当前锁定学科评定"（已在文案 §6 处理），避免用户误以为数据变了。

7. **旧列长期保留的技术债**：`journal_tier`/`tier_rank` 双轨保留是过渡权宜，长期两套"档位"概念并存有认知成本。v2 清理（建库停写、列 deprecated）需单独排期，本次不做。
