## 1. 目标与范围

### 1.1 要解决什么
- **F10**：用户能在「浏览」里把任意若干文献归入自建的「知识库分类」（跨 Zotero collection、跨 AI 主题的自由集合）。加入分类时，其中「有 PDF 且未深索」的成员**自动排队深索**——因为知识库分类的定位就是「我要深读/综合的一组文献」，深索是页级引用与综合的前提。
- **F11**：检索 / 对话 / MCP agent 能**限定在某个分类范围内**作答（把「分类」翻译成一组 key 白名单，喂给 `R.search`）。
- 两者共用一套数据模型（`kb_categories.json`）与一套「分类→keys」解析器，必须一起设计才自洽。

### 1.2 不含什么
- 不改检索算法本身（RRF/rerank/去重逻辑不动，只加 key 白名单过滤）。
- 不做分类的嵌套/层级（一期扁平即可；Zotero collection 的层级仍由镜像树承担）。
- 不做多用户/权限（单机单用户）。
- 不重建任何既有索引（纯增量：新加入分类的成员走既有选择性深索管线）。
- 拖拽入分类为**二期**（一期用右键菜单，保证多选可靠）。

## 3. 数据模型 / 存储

### 3.1 新文件 `data/categories/kb_categories.json`

```jsonc
{
  "version": 1,
  "categories": [
    {
      "id": "kbc_a1b2c3d4",          // "kbc_" + uuid4().hex[:8]，创建时定；rename 不变
      "name": "认罪认罚从宽",          // 展示名，可改，允许重名（id 才是主键）
      "source": "user",              // user=手建；ai=从 AI 主题固化而来（可选功能，见 §4.8）
      "keys": ["ABCD1234", "..."],   // 原始 Zotero item key，去重、保序（后加的在后）
      "note": "",                    // 预留备注，一期不填
      "created_at": "2026-07-08 10:00:00",
      "updated_at": "2026-07-08 10:12:00"
    }
  ],
  "updated_at": "2026-07-08 10:12:00"
}
```

**字段说明与设计取舍**
- `id` 用随机短 id 而非 name slug：rename 不破坏 F11 里已保存的 `category` 引用，也避免中文 slug 问题。前缀 `kbc_` 让 F11 解析器一眼区分「用户分类」vs「Zotero 路径」vs「topic」。
- `keys` 存**原始 key**：与 `zotero_collections.json.by_collection`、`papers.jsonl.key`、`extract.filter_scope` 全部对齐；深索/深索计数一律经 `is_deep()` 做 `safe_name` 映射（§4.2）。
- **不缓存 count/deep_count**：这两个数随「深索进度」「Zotero 删条目」漂移，**每次列举时现算**（成本：一次 `_load_papers()` 命中缓存 + 一次 `_deep_keys()` 读小文件，均 O(1)~O(篇数)，可接受）。
- 与既有数据关系：完全并列于 `zotero_collections.json` / `ai_topics.json` 的第三个 sidecar；不改它们，不进 LanceDB。孤儿 key（Zotero 删了的条目）在列举时被 `_load_papers()` 过滤掉，不显示、不计数，但保留在文件里（不主动清理，避免误删；可在 §4.7「移出」手动清）。

### 3.2 新文件 `data/state/deep_queue.json`（自动深索队列持久化）

所有新代码落在 `server.py`，紧邻既有 `/categories`、`/topics`、`/index/deep` 区块（约 `server.py:356-363` 与 `server.py:507-542`）。复用 `_load_papers()`、`_deep_keys()`、`_run_build()`。

### 4.1 并发安全的读写封装（新增，放在 `server.py:337` 附近）

```python
import uuid
_KBC_FILE = C.CATEGORIES_DIR / "kb_categories.json"
_KBC_LOCK = threading.Lock()        # 保护 kb_categories.json 的 RMW
_KBC = {"data": None, "mtime": 0}   # mtime 缓存，仿 _load_papers

def _kbc_load():
    """读 kb_categories（带 mtime 缓存）。缺文件→空骨架。"""
    if not _KBC_FILE.exists():
        return {"version": 1, "categories": [], "updated_at": ""}
    mt = _KBC_FILE.stat().st_mtime
    if _KBC["data"] is None or _KBC["mtime"] != mt:
        _KBC["data"] = json.loads(_KBC_FILE.read_text(encoding="utf-8"))
        _KBC["mtime"] = mt
    return _KBC["data"]

def _kbc_save(doc):
    """原子写：临时文件 + os.replace，避免半写。调用方须持 _KBC_LOCK。"""
    doc["updated_at"] = time.strftime("%Y-%m-%d %H:%M:%S")
    C.CATEGORIES_DIR.mkdir(parents=True, exist_ok=True)
    tmp = _KBC_FILE.with_suffix(".json.tmp")
    tmp.write_text(json.dumps(doc, ensure_ascii=False, indent=1), encoding="utf-8")
    os.replace(tmp, _KBC_FILE)
    _KBC["data"] = doc; _KBC["mtime"] = _KBC_FILE.stat().st_mtime

def _kbc_find(doc, cid):
    for c in doc["categories"]:
        if c["id"] == cid:
            return c
    return None
```

**并发规则**：所有改动走 `with _KBC_LOCK:` 内的「load→改→save」闭环。FastAPI 对 `def`（非 async）端点在线程池执行，多请求可能并发；`_KBC_LOCK`（`threading.Lock`）足以串行化。读端点（列举）也在锁内取一致快照后释放再算 count，避免与写交错。

### 4.6 加成员 —— `POST /kb/categories/{cid}/members`
入参 `{keys:[原始key]}`；出参把结果分三组回给前端做提示，并对 will_deep 入队：
```jsonc
{"ok":true, "added":["K1","K2"], "already":["K3"],
 "will_deep":["K1"],      // 有PDF+未深索 → 已入自动深索队列
 "no_pdf":["K2"],         // 无PDF，加入成功但不可深索
 "already_deep":["K4"],   // 已深索，直接可用
 "queued": 1}             // 实际入队数
```
```python
class KbMembersQ(BaseModel):
    keys: List[str] = []

@app.post("/kb/categories/{cid}/members")
def kb_cat_add_members(cid: str, q: KbMembersQ):
    papers = _load_papers(); deepk = _deep_keys()
    req = [k for k in dict.fromkeys(q.keys or []) if k]      # 去重保序
    added, already = [], []
    with _KBC_LOCK:
        doc = _kbc_load(); c = _kbc_find(doc, cid)
        if not c:
            return JSONResponse({"ok": False, "detail": "无此分类"}, status_code=404)
        have = set(c["keys"])
        for k in req:
            (already if k in have else added).append(k)
        c["keys"].extend(added)
        c["updated_at"] = time.strftime("%Y-%m-%d %H:%M:%S")
        _kbc_save(doc)
    # 分组：仅对「库中存在 + 有PDF + 未深索」的入队自动深索
    will_deep, no_pdf, already_deep = [], [], []
    for k in added:
        p = papers.get(k)
        if not p:                       # 孤儿 key：加了但不在库，忽略深索
            continue
        if is_deep(k, deepk):
            already_deep.append(k)
        elif p.get("has_pdf"):
            will_deep.append(k)
        else:
            no_pdf.append(k)
    queued = enqueue_deep(will_deep) if will_deep else []
    return {"ok": True, "added": added, "already": already,
            "will_deep": queued, "no_pdf": no_pdf, "already_deep": already_deep,
            "queued": len(queued)}
```
说明：`will_deep` 回给前端的是**实际入队**的（`enqueue_deep` 会再去重掉「已在队/在跑」的，见 §4.9），语义更准。

### 4.9 自动深索串行队列（核心，新增在 `_run_build` 之后 `server.py:531`）

```python
_DEEP_QUEUE_FILE = C.STATE / "deep_queue.json"
_Q_LOCK = threading.Lock()
QUEUE = {"pending": [], "in_flight": [], "timer": None}
_DEEP_BATCH = 50          # 每批最多深索篇数（避免一次拉太多、也让进度更细）
_DEEP_DEBOUNCE = 4.0      # 3–5s 合并节流

def _q_persist():          # 调用方须持 _Q_LOCK
    tmp = _DEEP_QUEUE_FILE.with_suffix(".json.tmp")
    tmp.write_text(json.dumps({"pending": QUEUE["pending"], "in_flight": QUEUE["in_flight"],
                   "updated_at": time.strftime("%Y-%m-%d %H:%M:%S")}, ensure_ascii=False),
                   encoding="utf-8")
    os.replace(tmp, _DEEP_QUEUE_FILE)

def _q_boot():             # startup 调用：回灌上次崩溃残留
    if _DEEP_QUEUE_FILE.exists():
        try:
            d = json.loads(_DEEP_QUEUE_FILE.read_text(encoding="utf-8"))
            # in_flight 可能已完成也可能没完成；回灌 pending，extract 幂等会跳过已完成的
            QUEUE["pending"] = list(dict.fromkeys((d.get("in_flight") or []) + (d.get("pending") or [])))
            QUEUE["in_flight"] = []
        except Exception:
            pass
    if QUEUE["pending"]:
        _drain_deep_queue()

def _queue_keyset():
    with _Q_LOCK:
        return set(QUEUE["pending"]) | set(QUEUE["in_flight"])

def enqueue_deep(keys):
    """入队一批 raw key：剔除已深索/已在队/在跑。返回真正入队的。触发防抖排空。"""
    deepk = _deep_keys()
    with _Q_LOCK:
        have = set(QUEUE["pending"]) | set(QUEUE["in_flight"])
        newk = [k for k in keys if k not in have and not is_deep(k, deepk)]
        if newk:
            QUEUE["pending"].extend(newk); _q_persist()
            if QUEUE["timer"]: QUEUE["timer"].cancel()
            QUEUE["timer"] = threading.Timer(_DEEP_DEBOUNCE, _drain_deep_queue)
            QUEUE["timer"].daemon = True; QUEUE["timer"].start()
    return newk

def _drain_deep_queue():
    """把一批 pending 转 in_flight 并起 deep build。撞锁则回滚，等 build 结束再排。"""
    with _Q_LOCK:
        QUEUE["timer"] = None
        if BUILD["running"] or not QUEUE["pending"]:
            return
        batch = QUEUE["pending"][:_DEEP_BATCH]
        QUEUE["pending"] = QUEUE["pending"][_DEEP_BATCH:]
        QUEUE["in_flight"] = batch; _q_persist()
    started = _run_build("deep", ["--scope", "keys:" + ",".join(batch)], on_done=_on_deep_done)
    if not started:                      # 撞了正在跑的 build（手动深索/更新）
        with _Q_LOCK:
            QUEUE["pending"] = batch + QUEUE["pending"]
            QUEUE["in_flight"] = []; _q_persist()

def _on_deep_done():
    with _Q_LOCK:
        QUEUE["in_flight"] = []; _q_persist()
    if QUEUE["pending"]:
        _drain_deep_queue()              # 继续排下一批
```

**改造 `_run_build` 支持 `on_done` 回调**（`server.py:507`）：
```python
def _run_build(stage, extra=None, on_done=None):
    if BUILD["running"]:
        return False
    def run():
        BUILD["running"] = True; ...            # 原逻辑不变
        finally:
            BUILD["running"] = False
            try: R.load_all(); BUILD["log"].append("[build] 索引已重载。")
            except Exception as e: BUILD["log"].append("[build] 重载失败：" + str(e))
            # 关键：任何 build 结束都尝试推进自动深索队列
            if on_done:
                try: on_done()
                except Exception as e: log_error("deep queue on_done", repr(e))
            else:
                try: _drain_deep_queue()   # 手动 build 结束后，接着跑排队的
                except Exception as e: log_error("deep queue drain", repr(e))
    threading.Thread(target=run, daemon=True).start()
    return True
```

**互斥关系梳理**（全部经同一个 `BUILD["running"]` 锁天然串行）：
- 自动队列 vs 手动「深索全库」/「更新」：谁先占锁谁先跑；后者结束时 `finally` 会 `_drain_deep_queue()` 接着排空自动队列。自动队列起 build 时若发现 `BUILD["running"]`，回滚 pending 等待。
- 多个分类连续加成员：`enqueue_deep` 各自把 key 塞进同一 `pending`，`_DEEP_DEBOUNCE=4s` 防抖把 4 秒内的多次拖动合成**一次** deep build，避免撞锁风暴。
- TOCTOU 说明：`BUILD["running"]` 的读在 `_Q_LOCK` 内、写在 build 线程内（无 `_Q_LOCK`），存在极小竞态窗口；真正的兜底是 `_run_build` 返回 `False` 时回滚 pending，正确性不依赖这个检查。低风险，单机可接受。

### 4.10 F11 分类→keys 解析器（新增，`server.py:363` 附近）

```python
def _resolve_category_keys(category):
    """把前端/agent 传的 category 值解析成原始 key 白名单 set；None/"" → None(不过滤)。
       支持三种前缀：kbc_（用户分类）/ topic:<id>（AI主题）/ zotero:<path>（收藏夹镜像）。"""
    if not category:
        return None
    if category.startswith("kbc_"):
        with _KBC_LOCK:
            doc = _kbc_load(); c = _kbc_find(doc, category)
        return set(c["keys"]) if c else set()          # 无此分类→空集(检索命中0，安全)
    if category.startswith("topic:"):
        try: tid = int(category.split(":", 1)[1])
        except Exception: return set()
        ait = _load_ai_topics()
        return {k for k, v in ait.get("by_key", {}).items() if v == tid}
    if category.startswith("zotero:"):
        path = category.split(":", 1)[1]
        return set(_load_cats().get("by_collection", {}).get(path, []))
    return set()      # 未知前缀→空集，不静默放行全库
```

**回归清单**
1. 不选分类：`/search`、`/chat`、`/papers`、MCP `search_localkb` 输出与改动前逐字一致（`keys=None`/`category=None` 分支）。
2. `/index/deep {scope:"all"}` 手动深索仍工作；结束后 `finally` 里 `_drain_deep_queue()` 不误触发（pending 空则 no-op）。
3. 单构建锁：手动深索进行中加成员 → will_deep 只入队不起 build；手动结束后自动排空。用 `GET /index/queue` 观测 pending→in_flight→清空。
4. key vs safe_name：造一篇 key 含非法字符（会被 `safe_name` 改写）的文献，加入分类深索后，分类列举 `deep_count` 正确 +1（验证 `is_deep` 全链路）。
5. 崩溃恢复：深索中途 kill server，重启后 `_q_boot()` 把 in_flight 回灌 pending 并续跑；`extract_one` 幂等 skip 已完成的（`extract.py:77`）。
6. 窄分类召回：30 篇分类做 F11 检索，返回条数接近 topk（否则按 §4.11 核实项调 fetch）。
7. 孤儿 key：Zotero 删掉某篇后重跑 light，分类列举把它排除出 count，不报错。

## 9. 风险与未决点

1. **窄分类召回衰减**（需核实，见 §4.11）：dense/bm25 各 50 候选，白名单过滤后小分类可能凑不满 topk。缓解已给（放大 fetch/`fused` 截断），但**是否够**要建库后实测。核实法：对已知规模分类跑 `/search` 数返回条数。
2. **TOCTOU on `BUILD["running"]`**（低风险）：队列检查锁与 build 状态锁不同源，靠 `_run_build` 返回 False 回滚兜底，正确性不依赖检查。单机可接受；若将来多 worker 需合并成一把锁。
3. **`threading.Timer` 防抖在打包环境的行为**（需核实）：daemon Timer 在 PyInstaller/nuitka 冻结版是否正常触发。核实法：分发包里连拖 3 篇间隔 <4s，确认合成一次 build。回退方案：去掉防抖，改为「每次 enqueue 直接尝试 `_drain`，撞锁则等 build 结束回调排空」——牺牲合并、但不依赖 Timer。
4. **孤儿 key 累积**：分类里 Zotero 已删的 key 不清理（只在列举时过滤）。长期看文件会留死 key。一期接受；可加一个「清理失效成员」维护动作（二期）。
5. **深索是全库共享、删分类不回收**：多个分类可能共享深索资产，故删分类/移出成员都不撤销深索。这是有意取舍——但意味着深索数据只增不减，磁盘只涨。若需回收得单独做 GC（超出本 epic）。
6. **`_run_build` 用 `subprocess` 跑 `build_all.py`**：自动队列每批起一个子进程，冷启动开销（加载模型等）按批摊薄——故 `_DEEP_BATCH=50` 不宜太小，否则频繁起子进程。值可调，需按实测 embed 速度定。
7. **对话限定分类 + 未深索**：分类里若全是未深索篇，full 模式只能召回 meta 行（且需语义档已建），召回质量低。已有文案兜底（§6），但产品上或应在下拉里对 `deep_count==0` 的分类给灰提示。未决：是否要禁选/降权此类分类——建议一期不禁，只文案提示。
8. **`category` 前缀协议是隐式约定**（`kbc_`/`topic:`/`zotero:`）：MCP agent 需从 `list_kb_categories` 拿 id，不能自造。已在工具描述里写明；风险是 agent 传错前缀 → `_resolve_category_keys` 返回空集（命中 0，安全失败，不会静默全库）。
