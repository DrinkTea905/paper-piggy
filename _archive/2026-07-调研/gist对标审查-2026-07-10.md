## 一、一句话结论

LocalKB 把 gist 的 **Query 一环做得比 gist 原文还完整**，但 **Ingest 一环完全没接**、**Lint 一环完全没写**。
现在的形态准确地说是：**一个很好的 RAG 检索系统 + 一个旁挂的答案缓存**，而不是 gist 说的
**「持续复利的活 wiki」**。差距不在代码质量，而在一条关键的接线从未存在。

gist 的三大操作实现度：

| 操作 | gist 要求 | LocalKB 现状 | 实现度 |
|---|---|---|---|
| **Ingest** | 导入一篇源 → LLM 读它 → 与你讨论要点 → 写摘要页 → 更新 index → 更新受影响的实体/概念页 → 追加 log。**一篇源触及 10–15 个 wiki 页** | 导入链路**一个 wiki 页都不碰** | ❌ 0% |
| **Query** | 检索 → 带引用综合 → **好答案回填成新页**，与导入的源一样复利 | 完整闭环，且超出要求（answer/concept/topic/digest/outline 五类页 + 页级引注 + 即时可检索） | ✅ 100%+ |
| **Lint** | 定期体检：矛盾、陈旧断言、孤儿页、缺失概念页、缺失交叉引用、数据缺口 | 完全不存在。没有路由、没有工具、没有 UI | ❌ 0% |

## 二、已经实现的（做得好的部分，不要动）

1. **第 1 层 Raw sources 的不可变性做得非常干净** ✅核实
   `config.py:5-9` 明写「所有写入落在 data/ 与 logs/，绝不写入知识库目录」；
   `zotero_source.py:52-56` 读 Zotero 时 `shutil.copy2` 出一份只读快照 `localkb_zotero_ro.sqlite` 再查，绝不回写。
   MCP 侧无任何改源/删源工具。这一层完全符合 gist 的 immutable 要求。

2. **答案回填 + 立即可检索的闭环，是全项目最亮的一块** ✅核实
   `/wiki/answer` → `save_answer` → `_persist_page` → `retriever.index_wiki_page`，
   写进同一张 LanceDB 表（`chunk_id = "{id}::wiki"`），**进程内即时可搜，不用重建索引**。
   三个入口都真接了线：网页「💾 保存此答案」、MCP `save_synthesis`、研究助手 digest/outline。

3. **沉淀的种类超出 gist 最低要求**
   不只沉淀对话答案，还有按需生成的 concept 页 / topic 页 / digest（带印刷页引注的资料汇编）/ outline（选题大纲），
   全部复用同一套持久化 + 入表管线，命中缓存 0 LLM 成本。

4. **provenance 命脉扎实**
   每页 frontmatter 带 `sources`（论文 key + 页级引用）/ `generated_by` / `generated_at` / `by_agent`，
   且页码由服务端**权威重解析**（`_resolve_citation`），不信客户端传来的引用。

5. **信任模型的骨架是对的**
   agent 写的页标 `by_agent=true` → 前端显示「🤖 综述页·未核验」；
   删除只走 `DELETE /wiki/page` + UI，**刻意不做成 MCP 工具**（`wiki_store.py:401` 有注释）。
   agent 能写不能删。

6. **删除页的三处联删是正确的**（这条是澄清，不是问题）
   `delete_page` 同时删 .md + index 条目 + LanceDB 表行，不会出现「页没了还能被搜到」的幽灵。

7. **没有「文档说做了、代码里其实是 TODO 占位」的情况**
   全库 grep 确认：差距全是「整块功能未写」，不是打桩糊弄。这一点值得肯定。

## 三、Wiki 页的真实状态（你最不确定的那块）

### 3.1 三个死字段 —— wiki 现在不是「图」，是一堆孤岛 ✅核实

`wiki_store.py:219` 里，每张新页都这样初始化：

```python
"stale": False, "by_agent": bool(by_agent), "links": [], "query": subject,
```

- **`links` 永远是 `[]`**。全库没有任何一行代码写入过非空 `links`。
  → gist 反复强调的「互链」「cross-references」「knowledge graph」，在 LocalKB 里**从未存在**。
  → 更糟的是每次 `regenerate` 还会把它重新清零。
  → 连带后果：orphan（孤儿页）检测无从谈起，因为**所有页都是孤儿**。

- **`stale` 永远是 `False`**。检索侧的降权机制全都接好了
  （`retriever.py:451` 的 `WIKI_STALE_PENALTY`、`config.py:132` 的配置值、`mcp_server.py:366` 的「⚠过时」徽标、前端徽章），
  但**写侧一根线都没有**——没有任何代码把 `stale` 置为 `True`。
  → 这是一台**永不触发的死机关**。用户以为「过时的综述会被标记降权」，实际上永远不会。

- **`by_source` 反查表建了却没人读**。`wiki_store.py:106` 有注释「供 stale 标记」，
  全库 grep 确认：**零消费者**。新增一篇论文，引用它的旧综述页不会被标脏，什么都不会发生。

### 3.2 缺失的页种

- **实体页（entity pages）不存在**。gist 至少两处并列点名 `entity and concept pages`，
  把作者/机构/案件/制度当作 wiki 的一等骨干节点。LocalKB 的 kind 只有 answer/concept/topic/digest/outline。
- **总览页 / 演进中的 thesis 不存在**。gist 视为灵魂的
  「an evolving synthesis / 随每次 ingest 被 strengthening or challenging 的全局论点」——没有。

### 3.3 三个真 bug（都已亲自核实）

| # | Bug | 位置 | 后果 |
|---|---|---|---|
| **B1** | **agent 可以静默覆盖你人工核验过的页** | `mcp_server.py:348` → `wiki_store.py:123` | `save_synthesis(title=…)` 传给 `save_answer(query=title)`，page_id = `sha1(title)`。**同标题 = 同 id = 无条件覆盖**。agent 用一个跟你已保存页相同的标题写回，你的页就没了，还被标成 `by_agent=true`。 |
| **B2** | **agent 可以让自己写的未核验页压过真论文** | `retriever.py:463-469` | `_apply_sort` 里只有 `blend` 分支调 `_blend_bonus`（内含 `_wiki_penalty`）。`relevance` 和 `tier` 两个分支**完全绕过降权**。而 `sort` 恰恰是 MCP `search_localkb` 开放给 agent 的参数（`mcp_server.py:78`，enum 含 relevance/tier）。→ 幻觉复利的直通车。 |
| **B3** | **index.json 是单点故障，且会自毁** | `wiki_store.py:76-82` + `_upsert_index` | `load_index()` 解析失败 → 静默返回空 `{"pages": []}` → 紧接着任何一次保存都会把这个空表 + 新页写回盘。**所有旧页的元数据永久裁掉**（.md 还在，但 index.json 是「权威事实来源」）。没有从盘自愈重建的能力。 |

补充：`index.json` 是无锁的读-改-写，`_atomic_write` 只防文件撕裂、不防丢更新。多 agent / 多会话并发写会丢页。

### 3.4 wiki 页的前端

- ✅ 好的：综述库好找、能读全文、有多个入口、空状态做得不错（0 页时有图标+引导，不是白屏）。
- ❌ **从 wiki 页看不到来源论文原文，更无法跳转 PDF 对应页**——点「来源」反而跳去「找相似」。
  这很致命：wiki 页的全部可信度都建立在「能一跳回溯原文」上，而这一跳是断的。
- ❌ 页间互链 / 反向链接 / 图视图在 UI 上完全不可用（因为 `links` 是空的）。
- ⚠️ 生成综述期间只有一个变灰按钮，没有进度/ETA/流式，长等待像卡死。
- ⚠️ `openWikiPage` 走 `jget`，又把人话错误吞回成「/path 404」（这是你上次修过的 `jpost` 同类问题，`jget` 上复发）。
- ⚠️ 「默认不存 + 主动保存」（网页对话）与「默认采纳 + 事后删」（agent 写回）**两套模型并存**，
  而且「🗑 不保存此答案」这个文案自相矛盾——它其实是「删除已保存的页」。

## 四、缺失的核心：Ingest 与 wiki 之间没有任何一根线 ✅核实

全库 grep `import wiki_store`，只有三个消费者：`server.py`、`retriever.py`、`research_assistant.py`。

导入链路——`folder_ingest.py`、`import_fulltext.py`、`extract.py`、`chunk.py`、`embed_index.py`、
`index_light.py`、`index_semantic.py`、`build_all.py`——**没有一个碰过 wiki_store**。

所以：

- 导入一篇新文献，触及的 wiki 页数 = **0**（gist 说 10–15）。
- 不新建 summary 页。`/index/deep_agent` 确实让 LLM 读了全文写了摘要，
  但摘要进了 `summaries.json` 当**嵌入前缀**用，不是 wiki 页。**形对而神偏**。
- 不更新任何已有的 concept/topic 页。
- 没有 `log.md`（gist 专门一节讲的 append-only 时间线）。
- 没有「与你讨论要点」的人在回路确认——现有的 human-in-loop 是**事后剔除**，不是**导入时讨论**。

这一条是整个差距的根源。gist 的立论是「**LLM 主动写、持续维护**」；
LocalKB 的 wiki 页**全部**是「用户点按钮 / agent 显式调工具」才写。
知识不会自己复利，它只在你想起来问的时候被动生成一页。

## 五、其他缺失

- **Lint 整条不存在**：无 `/wiki/lint` 路由、无 MCP 工具、无 UI 入口。
  矛盾检测、孤儿页检测、缺失概念页检测，一个都没有。
  （唯一部分等价物是 `suggest_new_sources`，但它的作用域是研究选题，不是 wiki 体检。）
- **wiki 不是 git repo**：没有版本历史、没有回滚。重生即永久覆盖，无后悔药。
  gist 说这是「白送」的，LocalKB 放弃了它——而它恰恰是「敢把写权交给 agent」的前提。
- **`index.json` ≠ gist 的 `index.md`**：没有每页一行摘要、非按类别组织。（这条是**有意偏离**，可接受。）
- **无 Marp / 图表 / canvas 输出**，改为 Word .docx 导出。（合理的领域偏移，法学场景 Word 更实用。）
- **frontmatter 缺 `tags`**，Dataview 只能半用。
- **Obsidian 互操作性差**：不是 `[[wikilink]]`，`data/wiki/` 不能当 vault 直接打开。
  （注：这是 gist 作者本人的核心工作流，但对法学教授价值存疑——见第七节评审意见。）

### 设计文档 vs 实现

- `实现设计_综合层与答案沉淀.md` 的 **Phase 0/1 完整兑现**，**Phase 2（矛盾候选提示）未做**（文档已声明可搁置，非隐瞒），
  **Phase 3（lint / stale / 增量监测）全链路缺失**。
- `实现设计_Agent工作流蓝图.md` 的 **11 条工作流仅约 2.5 条落地**，宣称的「波次一」只交付了 G2。
- **G1 页级引文核验器（`verify.py` / `verify_citation`）完全缺失**——
  它是蓝图里排名第一、被你自己称为「旗舰脊柱」的原语。
- 研究助手的 digest 写回**没有自动页级核验闸**，文档把它当红线 1 的核心防线，实际只靠 prompt 纪律。

## 六、用户友好（尚未解决的）

按严重度排序，只列此前几轮 UX 改造**没覆盖**的：

1. **「深索全库」一旦开始无法暂停或取消** —— 暂停键对整库深索被主动隐藏，子进程不可中断。
   可能空跑数小时并烧掉 API 额度。这是最严重的一条。
2. **无 key 时生成的「降级综述页」几乎没有质量警示**，反而在元信息里暴露程序黑话
   「模型 fallback(no-key)」。（你的 `index.json` 里真实躺着一页这样的 digest。）
   更糟：这个 fallback「综述」其实只是原文片段清单，却照样被嵌入进检索表、与真论文同表竞争。
3. **本地模式 1.2GB 模型下载中不可取消**，失败后只给「重试下载」，
   没有「切到 API 模式立即用」的逃生按钮。
4. **双击启动到窗口出现之间完全无反馈**（pythonw 无黑窗、无 splash），用户会反复双击。
5. 多处错误提示仍是程序员字符串。术语残留：「即时索引」「全文切块/向量化」。
6. `concept`/`topic` 综述在无 key 时直接 400 崩，与 digest/scope 的优雅降级不一致。
7. 主模态无法用 Esc 关闭、无焦点陷阱。
8. 破坏性操作有二次确认但**无撤销**（删综述页 = 不可逆的删文件 + 删索引）。

## 七、Agent 接入：三份独立提案 + 三位评审的裁决

三位评审（忠于 gist / 可实施性 / 用户价值）各自独立打分。**三票都进 top-pick 的只有两项**：

### 一致公认的第一优先（S 级改动、零回归）

| 项 | 是什么 | 为什么三票通过 |
|---|---|---|
| **server instructions** | `mcp_server.py:398` 的 `initialize` 返回里加一个 `instructions` 字符串字段，把 WIKI.md 的写回纪律直接下发 | 现状：agent 连上只从 `localkb_status` 拿到一个 WIKI.md **路径字符串**，永远读不到内容。gist 第 3 层 schema「让 LLM 成为纪律严明的 wiki 维护者」的命门，**从未被激活**。改一个字段。投入产出比最高。 |
| **read_source / get_source_text** | 让 agent 能读一篇论文的全文或指定页范围 | 现状：agent 只能拿到 220 字检索片段。`_extracted_excerpt`（`server.py:1668`）已经在逐页读 `config.EXTRACTED`，但**没有任何出口**。gist 的 ingest 主干「LLM reads it, extracts the key information」**根本走不通**。 |

### 各评审的分歧（这是需要你拍板的地方）

- **忠于 gist** 的评审力推：`propose_wiki_updates`（一篇源触及 N 页）、`update_wiki_page`、`lint_wiki`。
  他把「引注核验」压到 3 分，理由是「它是消费 wiki 的下游应用，不让 wiki 本身更像 gist」。
- **用户价值** 的评审力推：**引注核验（verify_citation + 核验一份带脚注初稿）打 9.5 分**，
  理由是「对法学研究者是杀手级刚需、投稿前自查、本地有中文期刊页级全文，云端做不到」。
  他把 Obsidian 原生化压到 4 分：「非技术法学研究者不用 Obsidian，那是 gist 作者本人的玩法」。
- **可实施性** 的评审力推：`mark_stale`、`list_sources`、`get_backlinks`——
  「读侧全接好、只差写侧一根线」「`by_source` 反查表已建好落盘、零消费者，加一个读出口就激活」。

### 三份提案共同的盲区（评审指出的）

1. **实体页（entity pages）没人提**，于是「导入一篇 → 更新受影响的实体页」在页种层面就无处落脚。
2. **演进式 thesis 总览页没人造**，wiki 停在「离散页 + 互链」，没有 gist 最看重的收敛论点。
3. **谁来「主动/持续」触发维护，三份都回避了**——这恰恰是 gist 的立论。
4. **接入门槛本身没人做**：对法学教授而言最高的墙是「怎么把 agent 连上」。
   而且三处文档口径（3 / 6 / 11 个工具）与真实的 **13 个**全对不上。
5. **只喂外部 agent，不管 app 内置对话**：非技术用户只活在 webapp 里。
   杀手级功能若只做成 MCP 工具，用户永远发现不了。
6. **没有成本 / 额度护栏**：verify、批量 ingest、单源触 15 页都在烧 API。
7. **双进程版本错配**：`mcp_server.py`（瘦客户端）与 `server.py`（守护进程）可能处于不同版本，
   没有能力协商 / 版本探测。新增 14 个工具会把文档漂移放大。
   → 建议让工具清单以 `TOOLS` 为**单一事实源**自动生成文档。

## 八、建议的三波次路线（待你拍板）

### 波次 0：地基止血（必做，与方向无关）
- 修 **B1**（save_synthesis 静默覆盖人工页）：agent 写回时若 id 已存在且 `by_agent=false`，拒绝或另起 id。
- 修 **B2**（relevance/tier 排序绕过 wiki 降权）：把 `_wiki_penalty` 提到 `_apply_sort` 的所有分支。
- 修 **B3**（index.json 自毁）：解析失败时不静默返回空，改为从 `.md` frontmatter 扫描重建 + 加写锁。
- 修「深索全库不可取消」。
- 修 no-key fallback 页被嵌入检索表这件事（它不是综述，是片段清单）。

### 波次 1：激活死代码（S 级，改动小、收益大）
- `server instructions`：把 WIKI.md 下发给 agent。
- `read_source`：暴露已存在的逐页全文。
- `mark_stale` + `get_backlinks`：`by_source` 反查表已建好，加两个出口就激活整套 stale 机制。
- `list_sources`：包装已有的 `GET /papers`。
- 文档以 `TOOLS` 为单一事实源自动生成，一次性解决 3/6/11/13 的口径漂移。

### 波次 2：接上 gist 的骨架（M–L 级，需你决定做到什么程度）
- `update_wiki_page`（按 id 建/覆盖/追加任意类型页）+ `set_wiki_links`（激活互链）。
- `propose_wiki_updates(source_key)`：导入一篇 → 返回「这篇影响了哪 N 个页、每页建议怎么改」。
  这是把 gist 的核心命题变成一次可执行调用的最小接线。
- `lint_wiki`：纯读 `index.json` 算 orphan / stale / source_changed，不碰检索，零回归。
- `git` 化 `data/wiki/`：agent 每次改都 commit，人能 diff / 回滚。
  这是**敢把写权交给 agent 的前提**，也是 gist 白送的东西。
- MCP `resources` / `prompts` 原语。

### 波次 3：法学场景的杀手锏（与 gist 正交，但用户价值最高）
- **G1 `verify_citation` + `verify.py`**：逐条判 supported / mismatch / not_in_lib 三态。
  （三份提案里 verify 独立出现三次 = 强信号。）
- 「检查我的引注」：粘贴稿件脚注 → 逐条核验报告。**必须做成 app 里的显眼按钮，不能只做 MCP 工具。**
- `save_synthesis` 加核验闸，从源头挡住无源 / 张冠李戴的断言落盘。
