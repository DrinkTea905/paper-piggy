## 一、gist 对标

**总结论：代码层面对标完成度约 80%，质量扎实；但真实运转度接近于零。** `data/wiki/` 目前只有 1 个真实页面，还是未配 LLM key 时的降级产物（诚实标注了"非合成综述"）；entities/concepts/answers/overviews 全空，git 历史里只有测试页写入又删除的痕迹。gist 的核心价值——复利累积——尚未开始发生。**最大缺口不在代码，在自动触发机制缺失 + 实际未投入使用。**

### 三环闭环打分

| 环 | 分数 | 现状 |
|---|---|---|
| Query（答案沉淀） | 7.5/10 | 最完整：保存→入检索表→即时可搜→三道防"幻觉复利"闸（综合时剔 wiki 证据、降级页不入表、检索降权+stale×0.3）。缺：沉淀纯靠手点，无主动建议 |
| Lint（体检） | 5.5/10 | 六类检查+UI+MCP 修复编排都有，断链还做到了写侧预防；**但 gist 明确要求的"定时自动跑"完全没有**——现有定时器只管索引不碰 wiki |
| Ingest（入库更新页） | 4.5/10 | 工具齐全（read_source/propose_wiki_updates/ingest-source prompt 把"一篇触 10-15 页"翻译成了可执行调用）；**但没有任何服务端扳机**：新文献深索后不会提示"影响了 N 个综述页"，全靠人工唤起外部 agent |

### 逐项清单（gist 设计点 → 实现状态）

- **三层结构** ✅ 已实现，schema（WIKI.md）比 gist 描述更细：版本号、防覆盖哈希、MCP initialize 时整篇下发
- **index** ⚠️ 有机器索引 index.json（含 by_source 反查、损坏时扫盘重建），无人类可读的 index.md；前端综述库列表+手写力导向关系图算功能替代
- **log.md** ❌ 未实现。git 提交历史（`page_id: 动作` 统一格式）是现成替代品，缺一个 `/wiki/timeline` 出口
- **frontmatter** ✅ 10 字段自足可重建；❌ 无 tags、**无 last_verified**（人工核验后不落任何时间戳，gist 点名的关键信息）
- **wikilinks** ⚠️ frontmatter links + backlinks + 图视图有；正文 `[[...]]` 语法不解析，正文内引用不做 drift 检测
- **新建页 vs 原地编辑启发式** ❌ WIKI.md 的 8 条写回纪律里没有这条规则，agent 行为会漂
- **git 版本管理** ✅ 超出 gist：MinGit 打包、退化快照兜底、UI 内历史/回滚
- **矛盾检测** —— 刻意不做（设计文档 §8 拍板：需 LLM 判断且规约禁止落成断言），属合理裁剪
- **hybrid search** ✅ 提前实现（bge-m3 + BM25 + RRF + reranker，wiki 行同表混检）

### 改进建议（按投入产出比）

1. **先让它转起来（零代码）**：配好 key，用真实文献跑一遍 `ingest-source` 全流程——当前所有结论都建立在测试页上
2. **lint 挂上现有定时器（~20 行）**：每天自动跑 `W.lint()`，首页角标"综述库有 N 处待理顺"
3. **Ingest 扳机（小改）**：深索完成回调里对新 key 调 `propose_updates`，非空则 UI 提示"《X》可能影响 N 个综述页"——把 Ingest 环从 4.5 拉到 7 分的那根线
4. **`/wiki/timeline` 时间线视图（低成本）**：git log 数据现成，即得 log.md 等价物
5. **last_verified 元数据（小改）**：人工确认页面时写 `verified_at`
6. **sources 相关性门槛**：沉淀时对 hits 设 reranker 分数下限再落 provenance（真实 digest 页已混入测试文件和离题文献）
7. **正文 wikilink 解析 + 断链 lint**
8. **WIKI.md 补"新建 vs 编辑"启发式**

## 二、检索质量

### 最重要的两个实态发现（比任何算法改进都关键）

1. **部署库 1398 篇全部只到 S 档（题录语义层）——深索 0 篇**：`LocalKB/data` 下 `extracted/`、`chunks/` 为空、无 embedded_keys.txt。"全文检索"名义存在、实际空转，所有命中都是标题+摘要级。
2. **`import_only_pdf: true` 把无 PDF 的法源挡在库外**：Zotero 里法条/判例多为 webpage/snapshot 无 PDF 条目，很可能整体没进索引——**法源 T1b 权重方案做得再好，召不回等于零**。

### 管线现状

- full 模式：bge-m3 稠密(top50) + jieba/bm25s(top50) → RRF(k=60) 等权融合 → 截池 ≤64 → 分类白名单过滤 → bge-reranker 重排 → 同 key 去重(MAX_PER_KEY=2) → 排序(relevance/tier/blend) → min_weight → topk
- 查询预处理只有 jieba 分词+停用词，**无查询改写/同义扩展/繁简归一**
- "开发者增强"= 期刊分级的个人档 profile（law_personal），不是分类机制；分类只做 key 白名单过滤，不参与打分
- **法源/报告权重方案已实施**（source_rules.py，注释标注"2026-07-12 用户拍板"）：三级定档 手动>itemtype>标题规则v3，法源 T1b=0.92、报告 T2=0.85，已接入权重解析/分级引擎/建库层。注意：papers.jsonl 是 07-08 生成的，早于规则实施，**需重跑 index_light 才能让建库层标签一致**（动态层已生效，低危）
- 期刊分级引擎（18 目录、学科感知、memo 缓存）工程质量高

### 发现的短板

- **权重尺度问题（老坑的残余）**：reranker 输出 0~10+ 的 logit，而 `WEIGHT_BONUS_SCALE=0.5` 意味着 T1(1.0) 与 T5(0.25) 在 blend 排序里只差 0.375 分——期刊/法源权重实际只起"近分决胜"作用，**提升幅度可能远低于预期**；同理新鲜 answer 页只减 0.05，而 answer 页标题≈用户原问题、reranker 打分虚高 3 分以上，**agent 写回的 answer 页会长期霸榜第一**（幻觉复利只堵了 stale 一半）
- **法条没有按"条"切分**：chunk.py 只有句边界+500 字贪心；引注模板也只有期刊两式，无"《××法》第 X 条"式
- **法律词典噪声**：2348 条全由 Zotero keywords 自动生成，混入"EXCEL技巧""《三体》"等；长短语成单 token 还会造成 BM25 查询错配
- **无时效因子**：被修订/废止的法条与现行版本同权混排，无标识——对法学是危险的
- 分类限定检索的候选池 bug 见 [bug 清单 #6](#五bug-清单)

### 改进建议（按法学检索提升幅度排序）

1. **把部署库深索真正跑起来**（零代码，运维+API 额度）——幅度最大
2. **放行无 PDF 的 statute/case/report 条目**（index_light 过滤处 3 行豁免 + 重建 L 档）
3. **法条按"条"切分 + 法规引注模板**（~100 行）
4. **修 answer 页霸榜**（kind=answer 改乘法折减或前端折叠为"已有综合"卡，~15 行+调参）
5. **轻量法学同义词表查询扩展**（少年司法↔未成年人司法等，人工百余组，bm25 侧 OR 扩展）
6. **分类过滤移到 RRF 截池之前 + 池上限联动放宽**（5 行调序，见 bug #6）
7. **法条版本/时效标识**（复用 wiki stale 机制）
8. **WEIGHT_BONUS_SCALE 实证校准**（抽 20 条法学 query 做金标集）
9. **词典治理**（按 itemtype 过滤+长度上限+静态法学核心词表）
10. 性能预案（深索普及后）：records 不存向量列、IVF-PQ、bm25 脏标记增量——当前不急

## 三、「最近入库」bug 根因

**已确认（高置信，代码+落盘数据双重佐证），这个功能从设计落地起就是坏的：**

证据链：
1. 前端「最近入库」卡 ← `GET /stats` ← `data/stats_cache.json` 的 recent 字段 ← index_light.py 重建时生成
2. `compute_stats` 取 **`papers[-8:]`（列表末尾 8 条）** 当"最近"（index_light.py:93-94）
3. 但 papers 顺序来自 zotero_source.py:70 的 SQL——**无 ORDER BY**，SQLite 走 (libraryID,key) 唯一索引按 **key 字母序** 返回。实测 papers.jsonl 2110 篇严格按 key 升序，stats_cache 的 recent 恰是字母序最末的 8 个 Z 开头 key（1995~2023 年的老文献）
4. 雪上加霜：`enrich()` 每次重建 **无条件把全库 ingested_at 重打成本次建库时刻**（index_light.py:66）——实测 2110 篇全是同一个值 `2026-07-08 13:43:18`，该字段完全不携带真实入库时间

所以：新文献进了 bm25/语义索引（检索能搜到✓），但除非它的随机 key 恰好排进全库字母序末 8 名（≈0.4% 概率），永远上不了「最近入库」。**连带受害**：卡片上「去『浏览』页看全部→」跳到的浏览页"按入库时间排序"也因全库同时间戳而无效（结果仍是 key 字母序）。

**修复方案（拍板项）**：Zotero items 表本身有 `dateAdded` 列——load_papers 时 SELECT 出来作为 ingested_at（最佳）；或重建时按 key 继承旧时间戳、只给新 key 打 now。然后 recent 改为按 ingested_at 降序取 8 条。顺手修：SQL 加显式 `ORDER BY i.itemID`；`/stats` 补字段那段 `except: pass` 加日志。

## 四、自动更新间隔

**当前默认 15 分钟，生效链条已全部定位：**

| 位置 | 值 | 性质 |
|---|---|---|
| `settings.py:29` `DEFAULT["auto_update"]["interval_min"]` | 15 | **唯一真正生效的默认**（load() 必经 _merge） |
| `server.py:131` 循环兜底 | 30 | 死代码，且与默认不一致（bug #24） |
| `server.py:165/175` 接口兜底 | 15 | 死代码 |
| `web/app.js:2323/2329` 前端兜底 | 15 | 死代码 |

**改成 1 小时的完整动作**：
1. `settings.py:29` 15→60（源码 + 按 SOP 同步到 app）
2. 同步把 server.py:131/165/175、app.js:2323/2329 的兜底统一成 60（顺手消灭三处不一致）
3. **关键**：部署版是便携模式（`LocalKB/portable.txt` 存在，数据在 `LocalKB/data`），其 settings.json **已经写入** `auto_update: {enabled: true, interval_min: 15}`——文件值优先于 DEFAULT，**必须同时改这份文件**（或改完代码后在设置页 UI 重设一次）
4. 循环每 60 秒醒一次检查、下限 max(5,…)，改 60 无兼容问题

## 七、Agent 接入

### 现状：基础出乎意料地好

- **21 个 MCP 工具**（13 读/8 写）+ 3 resources + 3 prompts，**描述质量优秀**（每个都回答"何时用我、参数从哪来、之后调什么"），initialize 时把 WIKI.md 规约整篇下发；`gen_mcp_doc.py` 防文档漂移
- 架构正确：MCP server 是瘦 stdio 客户端，一切经 HTTP 走主 server，无锁冲突；服务没起会自动拉起
- 写安全模型判断正确："能写不能删、写了可回滚"（409 拒覆盖人工页/强制 provenance/检索降权/MinGit 历史），不需要确认弹窗
- **FastAPI 的 /docs 与 /openapi.json 本来就开着**——OpenAPI 文档免费存在，只是没告诉任何人

### 主要弱点

1. **返回全是人读文本非结构化数据**：agent 要从 `«key:…»`、`[1] (T1)` 里反解析——写作型 agent 批量处理引注很吃亏（MCP structuredContent 未用）
2. **对照"agent 帮法学研究者写论文"全流程的缺口**：
   - 引注环节断在最后一米：**cite_format.py（G2 规则引擎）已建成却没有 MCP 出口**；G1 verify_claim 核验器未建（波次3被砍）
   - **agent 完全不能往库里加文献**（/ingest/files 存在未暴露）——suggest_new_sources 建议了缺失文献，却无法替用户放进库
   - 无单篇题录详情、无 /similar 出口、无"上次之后新进了什么"增量感知、无 wiki 页历史出口
3. list_wiki/list_sources 无分页；localkb.py CLI 停在 3 个功能的时代

### 增强路线（已含每项量级）

**波次 A（每项 ≤半天，纯暴露存量）**：`format_citation`（cite_format 出口）、`get_source_meta`、`similar_sources`、`whats_new`、`get_wiki_history`、RESOURCES 补声明 page template
**波次 B（1-3 天级）**：`locate_quote`（extracted 页文本字符级检索，~150 行，同时是 G1 地基）、`add_source`（本地路径入库，folder 模式，need_review 护栏）、structuredContent 结构化返回 + 分页
**波次 C（3-7 天级，蓝图 G1 旗舰）**：`verify_claim` 三态核验器（supported/mismatch/not_in_lib，铁律"库内无≠幻觉"）→ **Claude Code skill 包 `localkb-paper`**：把"意图澄清→迭代检索→grounded 提纲人工闸→分节起草绑源→逐条核验→排脚注→披露声明"编排成 SKILL.md（不新增服务端代码）→ G4 披露导出器（半天）

**接入方式结论**：MCP(stdio) 主通道维持；skill 包强烈值得做（论文工作流需要多阶段+人工闸，prompt 单段指令不够）；OpenAPI 零成本快赢（文档里写明 `http://127.0.0.1:8770/docs`）；CLI 建议文档明示"仅检索，完整能力走 MCP"+按需补 --read。

新工具的具体参数/返回设计详见调研原文（7 个工具逐一给出了 schema 级设计）。

部署目录 2.1GB = python 运行时 828MB + 两个 ONNX 模型 1.2GB + MinGit 91MB + **app 源码仅 4.3MB**。关键资产已就位：

### 推荐：路线 1

**macOS 现实**：Sequoia 起右键绕过 Gatekeeper 已移除，未签名 app 对非技术用户是显著劝退；正道 $99/年公证。**但 `curl | sh` 终端安装完全不触发 quarantine**——脚本安装路线在 macOS 可整体绕开签名。无 Mac 设备可用 GitHub Actions macos runner 构建。建议：先 Windows Setup.exe 验证需求，Mac 按需后置。
